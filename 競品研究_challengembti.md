# Challenge MBTI 競品研究報告

> 研究對象：<https://challengembti.com/>  
> 研究日期：2026-07-16  
> 研究方式：實際瀏覽首頁、分類頁、INTJ 挑戰入口、答題流程、失敗結果頁與分享選單，並檢查公開頁面 metadata、sitemap 與前端靜態資源。  
> 本報告用途：供後續產品與實作討論參考；目前未修改既有規格、程式碼或 Phase 狀態。

---

## 1. 執行摘要

Challenge MBTI 最值得參考的，不是 MBTI／星座題材，也不是「答錯死亡」本身，而是它如何把普通選擇題包裝成具有壓力、結果與分享衝動的完整傳播循環：

```text
挑戰集合 → 熱門挑戰 → 題庫入口 → 連續關卡 → 戲劇化結果 → 分享／重玩 → 相關挑戰
```

本專案目前的核心定位是「沒有標準答案的二選一，選完立即看真人 A/B 比例」。這個定位不應改成競品的「猜作者認定的正解」。較適合的做法是保留真人比例核心，借用競品的結果敘事、分享入口與題庫導流結構。

最值得納入 Phase 3 的單一建議是：在既有「你和 X% 的人品味相同」之外，增加一個更有記憶點的結果主標，例如「少數派冒險家」、少數派題數，以及「最孤獨的一題」。這些內容可以完全由可信的真人 A/B 統計推導，不需要虛構人格分數或設定正確答案。

## 3. 競品資訊架構

### 3.1 首頁

首頁分成兩個主要層級：

1. 挑戰集合：MBTI、星座、星宿關係、NPD 幸存挑戰。
2. 熱門挑戰：直接展示特定挑戰，如迴避型、天蠍座、INTJ、INFJ。

卡片不只顯示標題，也大量顯示社會證明與難度訊號：

- 挑戰人數。
- 通關率。
- 平均死亡關卡。
- 一句強烈的情境鉤子。

這讓使用者在進入題目之前，就先產生「真的這麼難嗎？」的好奇心。

### 3.2 分類頁

MBTI 分類頁把 16 型人格各自做成獨立挑戰，每個挑戰都有：

- 獨立名稱與隱喻，例如「前任防火牆」。
- 一句挑戰問題。
- 一段情境說明。
- 人數、通關率與平均死亡關卡。
- 獨立 URL。

它不是只把 16 個項目列成選單，而是讓每個項目都像一個可單獨傳播的產品。

### 3.3 個別挑戰入口

INTJ 挑戰入口先用一小段對話建立場景，再顯示：

- 挑戰名稱。
- 挑戰副標。
- 開始挑戰按鈕。
- 挑戰人數、通關率、平均死亡關卡。
- 相關挑戰。
- FAQ 與補充說明。

這一頁同時是開始頁、分享著陸頁與 SEO landing page。

### 3.4 答題流程

實測 INTJ 挑戰為 12 關，每題四個選項。畫面持續顯示：

- 目前關卡，例如「第 1/12 關」。
- 關卡名稱。
- 體面值。
- 覺察值。
- 執念值。
- 生存率。
- 當前關係情境與對話。

作答後直接進下一關，屬性數字立即改變。這些數字提供持續性的 stakes，使使用者感覺每一題都在影響同一場挑戰，而不是回答互不相關的問題。

### 3.5 失敗結果頁

實測在第 2 關選到高風險選項後，立即進入「死亡報告」：

- 「你死於第 2 關」。
- 死亡原因。
- 「67% 的挑戰者死在這裡」。
- 額外的復盤文案。
- 重新挑戰。
- 分享挑戰。
- 相關挑戰。

結果頁不是單純顯示分數，而是把結果寫成一句可以被記住、截圖與轉述的判詞。

### 3.6 分享入口

分享 modal 顯式列出：

- WhatsApp。
- LINE。
- Facebook。
- Threads。
- X。
- Reddit。
- LinkedIn。
- Email。
- 原生「更多」。
- 複製連結。

這比只有一個抽象的「分享」按鈕更能提醒使用者可以分享到哪裡。

## 4. 值得本專案參考的地方

### 4.1 把結果做成一句能被分享的身份描述

本專案目前規格的結果核心是：

- 每題顯示使用者選擇。
- 判斷多數派／少數派。
- 計算所選邊真人比例平均。
- 顯示「你和 X% 的人品味相同」。

資料定義是合理的，但「平均匹配度」本身較像統計摘要，不一定有足夠的分享衝動。

建議在結果第一屏增加：

```text
你是少數派冒險家
6 題裡有 4 題站在少數派
最孤獨的一題：只有 7% 的人和你一樣
```

下面再保留既有匹配度與逐題明細。

優點：

- 完全由既有真人票資料推導。
- 不需要建立人格模型。
- 不需要替題目設定正確答案。
- 比單一平均數更容易截圖與轉述。
- 可以形成自然的挑戰文案：「換你選看看？」

### 4.2 讓題庫本身有一致的主題與語氣

競品的題目因為共享一個 Boss／情境，因此即使互動形式簡單，仍有連續感。

本專案不用加入分支劇情，但 Phase 4 的每個題庫可保持：

- 一致的核心問題。
- 一致的文案語氣。
- 題目順序有節奏，而非任意排列。
- 開場題簡單、中央題最具爭議、最後題適合作為結尾。

例如題庫不只叫「生活選擇」，而可以有一句較具挑戰性的承諾：

> 這些日常選擇，你真的和大家一樣嗎？

這不改變 A/B 比例的產品本質，但能讓一組題目更像一次完整體驗。

### 4.3 在入口卡片展示有意義的真實訊號

競品使用通關率與平均死亡關卡快速表達難度。本專案沒有通關概念，不應硬造同類數字，但可以在資料可靠後研究：

- 熱門：真人參與 session 達到門檻。
- 最多人分歧：題庫內接近 50/50 的題目比例較高。
- 最出乎意料：存在明顯少數選項，但文案容易誘發直覺誤判。
- 本週熱門：依可信的每日參與彙總計算。

任何標籤都必須有明確、可重算的定義，並受低樣本門檻保護。

### 4.4 顯式提供台灣常用分享目的地

本專案已規劃 Web Share API 與複製連結 fallback，技術方向正確。競品提醒了一個 UX 問題：使用者看到具體平台名稱時，分享行為更明確。

Phase 3 可評估顯式顯示：

- LINE。
- Threads。
- 複製連結。
- 更多（Web Share）。

分享內容仍應指向題庫首頁，不必建立動態結果 URL。使用者若要展示個人結果，主要方式仍是截圖總結卡。

建議分享文字：

> 我有 4/6 題站少數派，最離群的一題只有 7% 跟我一樣。換你選看看？

實作前需確認不同平台的分享 URL 支援狀態；不應只因競品列出某平台，就假設其分享端點長期穩定。

### 4.5 結果後立即導向下一個題庫

本專案 Phase 3 已規劃「你可能也想玩」兩張其他題庫卡，方向正確，應保留。

可以再注意：

- 不要只隨機挑選；至少避免當前題庫。
- 卡片文案要提供不同的選擇衝突，而不只換封面。
- 導流應在結果主卡與分享動作之後，避免搶走分享焦點。

### 4.6 每個題庫都是獨立著陸頁

競品替首頁、分類頁與個別挑戰建立獨立 URL，並包含：

- title 與 description。
- Open Graph metadata。
- Twitter large image card。
- canonical／語言 alternate。
- sitemap。
- Breadcrumb、FAQ、WebApplication／Game 等 JSON-LD。

本專案 Phase 3 已規劃 per-quiz 靜態頁與 OG，這是正確方向。Phase 5 可額外評估：

- sitemap.xml。
- canonical URL。
- 基本 WebSite／WebApplication 或 Breadcrumb JSON-LD。
- 每個題庫頁有實質、非重複的簡介。

多語系目前不在本專案首版範圍，不建議因競品有多語就擴張 Phase 3。

## 5. 不建議照抄的地方

### 5.1 不要加入有正確答案的生存／死亡判定

競品的魅力來自「活到第幾關」，但代價是選項背後存在作者認定的好壞與扣分規則。

本專案的價值是：

- 沒有標準答案。
- 讓使用者比較自己與真人群體的選擇。
- 爭議本身就是內容。

若加入死亡、失敗或正確答案，會改變產品契約，也容易讓輕鬆題材變成被評判的體驗。

### 5.2 不要加入體面值、覺察值、執念值等多軸分數

這些屬性很適合競品的關係模擬遊戲，但本專案若照抄，必須額外定義：

- 每個選項如何加減分。
- 分數是否有心理學含義。
- 結果類型如何產生。
- 如何避免使用者誤解為人格診斷。

這會讓題庫製作與驗證成本大幅增加，也偏離最小可行產品。

### 5.3 不要把來源不清楚的數字當作社會證明

公開前端資源中可以看到類似以下欄位：

- `globalBase: 318904`
- `passRate: "0.7%"`
- `averageDeath: "第2關"`
- 每關預設 `deathRate`

主要挑戰前端程式中未觀察到即時統計 fetch；上述數字看起來至少包含預設靜態內容。因此，不能把競品畫面上的「全站數據」視為其已證明具有可靠的真人統計架構。

這點反而凸顯本專案 Phase 2.5 的價值：真人票、參與數、完成數、分享數與 page view 必須分開定義，公開數字必須可追溯、可重算且受低樣本保護。

### 5.4 不要在首版加入多語、廣告與付費內容

競品目前具備多語入口、廣告與其他商業化路徑，但這些都不是本專案目前驗證核心體驗所需。過早加入會：

- 增加內容維護成本。
- 增加 privacy／consent／第三方 script 複雜度。
- 讓答題頁分心。
- 延後可信統計、結果卡與正式題庫上線。

### 5.5 不要為 SEO 填入大量重複文字

競品在挑戰頁下方加入長篇補充說明與 FAQ。獨立著陸頁有文字內容是合理的，但本專案應以實質資訊為主：

- 這個題庫在選什麼。
- 大約需要多久。
- 題數。
- 統計資料如何理解。
- 娛樂性與非診斷性聲明（若題材需要）。

不要用大量同義改寫干擾主要答題體驗。

## 6. 對各 Phase 的具體建議

### Phase 2.5 — 公開統計可靠性

維持既有規格，不因競品的大數字而降低標準：

- 公開 A/B 比例只計後端已接受的真人有效票。
- 顯示有效樣本數。
- 參與、完成、分享與 page view 分開命名。
- 低於門檻不顯示人數或衍生標籤。
- 未來若增加「熱門」「最分歧」，需先寫出精確公式與時間窗。

建議先完成 Phase 2.5，再設計依賴公開統計的結果文案；否則 Phase 3 容易先建立無法兌現的數據承諾。

### Phase 3 — 總結卡與分享

建議在開工展開實作契約時討論：

1. 結果主標如何由資料決定。
2. 少數派題數的定義（50% 是否算少數派需明定）。
3. 「最孤獨的一題」同票／零樣本處理。
4. 低樣本題是否排除結果敘事。
5. 結果卡首屏、逐題摘要與導流的資訊層級。
6. LINE／Threads／複製連結／Web Share 的能力偵測與 fallback。
7. 分享成功事件何時回報，避免把開啟 modal 當成成功分享。
8. per-quiz OG 圖片、title、description 的產出方式。

建議維持「分享題庫首頁連結、個人結果靠截圖」的現有決策，避免引入動態結果頁與伺服器產圖。

### Phase 4 — 題庫內容

每個正式題庫可新增以下內容設計要求：

- 一句可放在封面／OG 的挑戰鉤子。
- 題目排序有起承轉合。
- 各題衝突維度不要重複。
- 至少一題有機會形成高度分歧。
- 結尾題適合在結果前收束體驗。
- 文案避免暗示任何選項是道德上正確答案。

這些應寫入個別 `subdocs/題庫/<id>.md`，不必讓共用 JSON schema 承擔過多遊戲化欄位。

### Phase 5 — 上線與發現性

除既有部署與正式分享實測外，可評估：

- sitemap.xml 是否涵蓋所有題庫著陸頁。
- canonical URL。
- robots.txt。
- 基本 JSON-LD。
- OG 圖片 1200×630 與實際平台裁切。
- LINE、Threads、Facebook 的真網域預覽。
- Cloudflare Web Analytics 與第一方統計命名是否清楚分離。

## 7. 建議優先順序

1. **先完成 Phase 2.5**：確保所有公開真人統計可信。
2. **Phase 3 強化結果主標**：加入少數派題數與最孤獨的一題。
3. **Phase 3 強化分享入口**：顯式提供 LINE、Threads、複製連結與原生更多。
4. **Phase 4 強化題庫敘事一致性**：不加入正解，只提高題庫整體感。
5. **Phase 5 補發現性基礎**：sitemap、canonical、基本 JSON-LD 與各平台 OG 實測。

不建議為了競品對照而插入新的大型 Phase。以上大多可在既有 Phase 2.5、3、4、5 的實作契約展開時吸收。

## 10. 最終判斷

Challenge MBTI 證明了一件事：趣味選擇題的傳播力不只來自題目，而是來自「我得到了什麼可轉述的結果」。

本專案已經擁有更適合長期信任的基礎——清楚定義的真人 A/B 統計。下一步不應把它改造成另一個人格闖關遊戲，而應把可信統計翻譯成更有情緒張力、可截圖、可挑戰朋友的結果敘事。

建議保留產品核心，只吸收三件事：

1. 每個題庫有一句鮮明的挑戰承諾。
2. 每次完成都有一句可記住、但可由真人數據證明的結果主標。
3. 分享後能自然導向朋友開始玩，以及使用者自己玩下一個題庫。
