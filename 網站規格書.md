# 二選一測驗網站 — 規格書

繁體中文「Would You Rather」風格趣味測驗網站。使用者從題庫中逐題二選一,點選後即時看到全站選擇比例,答完產生可分享的總結卡。所有題庫由站方(擁有者)自訂,無使用者建立功能。

> **文件分層**:本檔為**意圖層**——系統該有什麼行為、什麼狀態必須為真(各節「驗收」段)。實作契約(檔案清單、API 細節、實作決策)見 `開發設計方針.md`;click-by-click 驗收步驟見 `測試指南.md`。

---

## 1. 技術架構

| 層 | 選型 |
|---|---|
| 前端 | 靜態網站,部署於 Cloudflare Pages(框架不限,輕量為佳) |
| 後端 | Cloudflare Workers |
| 資料庫 | Cloudflare D1(SQLite) |
| 分析 | Cloudflare Web Analytics(免 cookie) |
| 防護 | Cloudflare WAF rate limit(免費方案:單一規則、10 秒計數窗、非精確閘門——定位為**減速丘**,不宣稱防灌票;進階加固見 §5.4 暫緩預案) |

動態部分極小:一張資料表、三支 API,其餘全靜態。(公開統計加固預案見 §5.4,**暫緩、條件觸發**。)

## 2. 頁面結構

### 2.1 入口頁 `/`
- 頂部:分類 chips(全部/生活/美食/宅宅…),可橫向滑動,點選過濾題庫。**初期以 feature flag 隱藏**(題庫少時不需要),程式先寫好。
- 題庫列表:**單欄大卡片**,每張卡含封面圖(16:9)、標題、題數、「N 人玩過」(來源:該題庫第一題 a+b 總票數,**含種子票**;見方針 Phase 2 決策)。
- 「N 人玩過」低於門檻(預設 100,可設定)時不顯示。

#### 驗收
- manifest 中所有題庫以單欄大卡片呈現,含封面、標題、題數。
- 「N 人玩過」達門檻才顯示;低於門檻的題庫卡完全不出現該行。
- flag off 時分類 chips 完全不渲染;flag on 時點選 chip 正確過濾列表。

### 2.2 題庫首頁 `/quiz/:slug`(分享著陸頁)
- 內容:大封面圖、標題、一兩行簡述、大「開始作答」按鈕、小字「約 1 分鐘 · N 題 · 已有 N 人作答」(同樣受門檻限制)。
- **必須有完整 OG meta tags**(og:title、og:description、og:image 用封面圖)——分享到 LINE / Threads / FB / IG 時預覽卡是傳播命脈。
- URL 使用有意義的英文 slug(如 `/quiz/daily-life`),不用數字 id。
- 按下「開始作答」時一次抓取該題庫全部統計(GET /api/stats/:quizId),之後比例計算全在前端。

#### 驗收
- 每個題庫首頁的 OG meta 完整(og:title / og:description / og:image 為封面圖絕對網址),分享預覽卡正確。
- 「開始作答」只發出一次 stats 請求;整個答題流程中不再有 stats 請求。
- 「已有 N 人作答」受同一門檻限制。

### 2.3 題目頁(答題流程,SPA 內切換)
- 手機直式**上下對決式**版面:上選項卡 / 中間 VS 分隔帶 / 下選項卡。
- 每張選項卡:插圖(固定比例,建議 4:3 或 1:1)在上、文字在下。
- 頂部細進度條 + 「3/16」文字。
- **無返回上一題功能**,單向流動。

(互動驗收見 §3。)

### 2.4 總結卡(答完顯示)
- 不含選項插圖。內容(由上而下):
  - **結果主標區(首屏截圖視覺焦點,2026-07-16 定案)**:
    - 身份標籤(大字)+ 副標「你在 X/N 題站在少數派」。
    - 標籤四檔,依**少數派題數佔比**判定(題庫題數不定,用佔比不用絕對值):

      | 少數派佔比 | 標籤 |
      |---|---|
      | ≥ 50% | 逆流獨行俠 |
      | 25%–49% | 少數派冒險家 |
      | 1%–24% | 有主見的主流派 |
      | 0 | 大眾品味代言人 |
    - **最孤獨的一題**:候選 = 使用者站少數派的題(所選邊顯示比例 < 50%;**恰 50% 不算少數派**);取所選邊比例最低者,同率取題序在前;顯示題目與「只有 X% 的人和你一樣選 OO」。全部站多數派時改顯示「你最大眾的一題:X% 的人都和你一樣」,不留空白。
    - 所有比例口徑與答題頁一致(開場快照 + 自己一票),截圖對照不出現兩套數字。
  - 每題摘要:你的選擇 + 你是多數派/少數派。
  - 匹配度數字:取使用者所選那邊的全站比例平均,顯示「你和 X% 的人品味相同」(保留,置於主標區之後為次要資訊)。
  - 版面設計成適合手機截圖的滿版畫面。
- 分享鍵:優先 Web Share API(`navigator.share`),**分享文字帶結果摘要**(例:「我在 6/15 題站少數派,最孤獨的一題只有 7% 和我一樣。換你選看看?」)+ **題庫首頁連結**;不支援時(IG/Threads in-app browser 常見)fallback 為「複製連結」按鈕,複製內容為同樣文字 + 連結。不建動態結果頁、不做伺服器產圖——個人結果的展示方式是截圖。
- 下方導流:「你可能也想玩」顯示 2 個其他題庫卡片。
- 頻道連結以一行小字低調呈現。

#### 驗收
- 身份標籤依少數派佔比四檔正確切換;副標 X/N 與逐題明細的少數派標記數一致。
- 「最孤獨的一題」取少數派中所選邊比例最低者,同率取題序在前;恰 50% 不計入少數派;全多數派顯示「你最大眾的一題」變體。
- 主標區所有比例與答題頁口徑一致(快照 + 自己一票)。
- 每題摘要的多數派/少數派判定與比例一致;匹配度 = 所選邊全站比例的平均,整數顯示。
- 支援 `navigator.share` 的環境分享「結果摘要文字 + 題庫首頁連結」;不支援時顯示「複製連結」且複製內容為同樣文字 + 連結。
- 導流區塊顯示 2 個**其他**題庫(不含當前題庫)。
- 直式手機下總結卡為滿版、適合截圖,無選項插圖。

#### §2.4 擴充:孤獨稱號同框展示(2026-07-16 定案;屬 Phase 3.5,不在 Phase 3 驗收範圍)

- **概念**:四檔身份標籤講「量」(整體多反骨),孤獨稱號講「質」(最反骨的那件事),同框分層展示。
- **版面**:身份標籤 + 副標維持主標地位不變;「最孤獨的一題」區塊升級為**勳章元件**——稱號(勳章名)+ 原有題目與「只有 X% 的人和你一樣選 OO」+ 判詞一行。
- **資料**:稱號由最孤獨的一題的 `questionId` 查表取得;計算規則(恰 50% 不算少數派、同率取題序在前、全多數派變體)完全沿用 §2.4 既有定義,**不新增任何統計口徑**。
- **稱號表為題庫內容**,放題庫 JSON 選配欄位:`titles: { <questionId>: {name, blurb}, mainstream: {name, blurb} }`;全多數派時 `mainstream` 稱號配「你最大眾的一題」變體。內容定稿於各 `subdocs/題庫/<id>.md`。
- **降級**:題庫無 `titles` 欄位時勳章不顯示,版面回到 §2.4 原樣(demo 等舊題庫零改動)。
- **分享文字**帶稱號(例:「我是『少數派冒險家』,還拿到『糖度異端』稱號——只有 7% 和我一樣。換你選看看?」);無稱號時維持原分享文字格式。

##### 驗收(Phase 3.5)
- 勳章稱號與最孤獨的一題的 questionId 查表一致;全多數派顯示 `mainstream` 稱號。
- 無 `titles` 題庫不顯示勳章、不報錯,版面與 §2.4 原規格一致。
- 分享文字含稱號且與畫面顯示一致;無稱號題庫的分享文字維持原格式。

## 3. 答題互動(核心體驗)

1. 題目載入時,兩邊選項卡**皆不顯示比例**。
2. 使用者點選任一邊:
   - 該邊立即進入選中狀態(主色框、放大強調),另一邊降透明度但**仍可點選**。
   - 兩邊比例條播放**變長動畫**,動畫結束(`transitionend`)後才淡入百分比數字。
   - **動畫每題只播一次**。之後改選只瞬間切換選中狀態與數值,不重播動畫。
3. 比例 = 「開始作答時抓的統計快照 + 使用者自己這一票」,前端即時計算,零延遲。
4. 按「下一題」的瞬間,才將**最終選擇**寫入後端(見 §5),然後切至下一題。
5. 可選優化(實作後手感測試決定):點選後 2 秒無操作自動進入下一題,期間仍可改選。

### 比例顯示規則
- 百分比只算一邊,另一邊用 `100 − x`,避免四捨五入後加總 ≠ 100%。
- 整數顯示。

### 驗收
- 載入時無比例;點選後才出現,且動畫結束才淡入數字。
- 動畫每題僅播放一次;改選為瞬間切換,無動畫重播。
- 改選後另一邊仍可再點,選中狀態與數值即時正確。
- 比例含使用者自己一票,兩邊加總恆為 100,整數顯示。
- 只有按「下一題」才送出投票;改選過程零網路請求。
- 無任何返回上一題的途徑;進度條與「n/N」隨題推進正確。

## 4. 題庫資料格式(靜態 JSON)

### 4.1 `manifest.json`(入口頁載入)
```json
{
  "quizzes": [
    {
      "id": "daily-life",
      "title": "日常生活二選一",
      "cover": "/img/daily-life/cover.webp",
      "category": "life",
      "questionCount": 16
    }
  ]
}
```

### 4.2 題庫檔 `/quizzes/:id.json`
```json
{
  "id": "daily-life",
  "title": "日常生活二選一",
  "description": "咖啡還是茶?看看你和大家的選擇差多少",
  "cover": "/img/daily-life/cover.webp",
  "category": "life",
  "questions": [
    {
      "id": "coffee-tea",
      "a": { "text": "喝咖啡", "img": "/img/daily-life/coffee.webp" },
      "b": { "text": "喝茶",   "img": "/img/daily-life/tea.webp" }
    }
  ]
}
```

規則:
- `question.id` 為有意義的固定字串,**永不因排序調整而變**;統計以 `quiz_id + question_id` 為 key。
- 題目順序固定(由站方設計節奏),不隨機。
- 圖片:WebP、正方形或 4:3、每張 30–50KB。全題庫風格一致(以固定風格 prompt 模板生成,見 `插圖風格指引.md`,Phase 4 建立)。

## 5. 後端

### 5.1 資料表
```sql
CREATE TABLE stats (
  quiz_id     TEXT,
  question_id TEXT,
  a_count     INTEGER DEFAULT 0,
  b_count     INTEGER DEFAULT 0,
  PRIMARY KEY (quiz_id, question_id)
);
```
注意:**不可用 KV 做計數**(每 key 每秒僅一次寫入,會掉票);D1 的原子 UPDATE 才正確。

### 5.2 API

**GET `/api/stats/:quizId`**
回傳該題庫所有題目的 `{ questionId: { a, b } }`。

**POST `/api/vote`**
- body:`{ "quizId": "...", "questionId": "...", "choice": "a" | "b" }`
- 後端以 UPSERT 原子遞增對應計數。
- **必須驗證 quizId / questionId 存在**(Worker 內建合法清單),否則拒絕,防止垃圾 rows。
- 前端送票不阻塞 UI(Phase 2.6):票先記入 localStorage `pendingVotes`,以 `fetch(..., { keepalive: true })` 背景送出、不等回應即切題;收到成功回應才清除 pending;失敗留存,開站與恢復連線(`online`)時補送,超過 7 天丟棄,同題同時最多一個 in-flight。
- 後端冪等(Phase 2.6):vote 帶前端隨機產生的匿名 `sessionId`;D1 收據表以 `(session_id, question_id)` 唯一鍵去重,收據插入與計數遞增在同一交易;重複送達回成功但不加票。sessionId 現階段不做驗證,僅作冪等 key——這是 §5.4 預案的插座:日後觸發時只需在發 session 處加 Turnstile 閘,表與 API 不變。

### 5.3 種子票
- 新題庫上線時,每題預填種子票:**兩邊合計約 100 票左右**,隨機分佈,避免早期「100% vs 0%」的尷尬。
- 各題實際分佈於建題時逐題人工決定(以 seed script 執行)。
- **定位(站方決策,2026-07-16)**:本站是趣味測驗不是民調,種子票視為統計的一部分,比例與「N 人玩過」皆含種子票;若日後啟動 §5.4 預案,再改為種子 / 真人分離。

### 驗收
- 非法 quizId / questionId 的 vote 被拒絕,資料表不產生垃圾 rows。
- 投票為原子遞增;並發投票不掉票。
- vote 失敗(斷網、被拒)不影響前端 UI 與答題流程。
- 暫時失敗的票最終送達;同一 `(session_id, question_id)` 無論重送幾次只計一次(Phase 2.6)。
- 種子票 script 執行後,各題兩邊合計約 100 票、隨機分佈。

### 5.4 公開統計可靠性(Phase 2.5)——**暫緩,條件觸發**

> **狀態(2026-07-16 站方決策)**:本節整體為預案,不排入開發順序、不阻擋 Phase 3。
> **觸發條件(任一)**:單題單日票數增量超過 Web Analytics 同日頁瀏覽量 3 倍;任一題比例在短時間內劇烈翻轉;或流量成長到站方認為公開數字可信度開始重要。
> **觸發時的啟動成本**:Phase 2.6 的收據表已以 `(session_id, question_id)` 為唯一鍵,只需在發 session 步驟加上 Turnstile + Siteverify 閘與簽章,資料表與 vote API 不變。
> 觸發前,§2、§5.2、§5.3、§6 一律以各節正文為準,本節不覆寫任何章節。

#### 統計語意

- 公開 A/B 比例只計算後端已接受的真人有效票;畫面同時顯示該題有效樣本數。
- 「N 人玩過」= 已成功送出至少一票的唯一匿名 session 數。
- 「完成數」= 該匿名 session 的合法題目全部有一筆已接受選擇;同一 session 只計一次。
- 「分享數」= 使用者觸發分享 / 複製連結且前端回報成功的互動次數,不是社群平台實際曝光數。
- Cloudflare Web Analytics 負責 page view / referrer;題目票數、開始、參與、完成、分享與每日趨勢
  由第一方後端彙總,兩者不可混稱。

#### 匿名 session 與防重

- 按「開始作答」時先取得 Turnstile token,由後端 Siteverify 驗證成功後建立題庫限定的匿名 session;
  client-side widget 成功但未經後端驗證不得視為有效。
- session 使用不可猜測 id + 後端簽章 token,綁定 quizId 與有效期限;不儲存 IP、User-Agent 或瀏覽器指紋。
- 同一 session 對同一 questionId 最多接受一票。相同內容重送回成功(冪等),不同 choice 重送回衝突,
  不得改寫已計入的票。
- localStorage 仍只作使用體驗與降噪;真正的防重邊界在 D1 unique constraint / transaction,不能信任前端。
- Phase 5 以 WAF rate limit 保護 session / vote 寫入路徑;rate limit 是緩解濫用與過載,不是
  「一人一票」證明,門檻需依正式方案與壓測結果設定。

#### 可靠送票

- UI 選擇與切題維持零等待;按「下一題」時先把 vote 放入本機 pending queue,再背景送出。
- 只有後端回覆 accepted / duplicate accepted 後才從 pending queue 移除;網路錯誤、429、5xx 可在退避後
  重試,重開頁面或恢復連線時亦會補送。
- 後端必須以 sessionId + questionId 做冪等,讓重試不會重複加票。
- session 建立或統計快照讀取失敗時不得用空物件繼續答題;顯示可重試錯誤。已開始後送票失敗則不阻塞
  作答,但需顯示非阻塞的「統計同步中」狀態。

#### 資料與營運

- D1 分開保存:累積真人 A/B 計數、匿名 session / 每題收據、題庫累積指標、每日題庫指標與每日題目 A/B。
- 每題收據與累積計數必須在同一交易邊界更新;任何可見成功回應不得只完成其中一邊。
- 原始匿名 session / 收據只保留完成防重與除錯所需期限;長期趨勢以不含 sessionId 的每日彙總保存。
- production 上線前要有 D1 Time Travel / 匯出與還原 runbook,並記錄 Workers / D1 用量告警與升級門檻。

#### 驗收

- 相同 session 同題並發或重試多次,真人計數恰好 +1;不同 choice 重送不改票。
- 模擬斷網 / 429 / 5xx 後恢復,使用者流程不中斷,pending 最終送達且不重複計票。
- 任意偽造、過期、quizId 不符的 session token 與未通過 Siteverify 的 session 建立請求皆被拒絕。
- production 新題庫從 0 真人票開始;公開比例、匹配度與玩過人數均不含 staging 假資料。
- participant / completion / share 與每日彙總在重試後仍各自只增加一次,定義與畫面文案一致。
- 受控 burst 測試中,成功接受數與 D1 增量一致;429 / 5xx 保留在 client pending queue,無 silent loss。

## 6. localStorage 與重複計票

localStorage 結構(粒度到**題**):
```json
{
  "daily-life": {
    "answers": { "coffee-tea": "b", "wine": "a" },
    "pendingVotes": { "wine": { "choice": "a", "queuedAt": 0 } },
    "completedAt": null
  }
}
```

規則:
- 每題送票前檢查:該 `questionId` 已有紀錄 → **不送 POST**(重玩、續玩皆同)。
- pending 票於開站 / 恢復連線時補送;後端以 `(session_id, question_id)` 冪等,補送不重複計票(§5.2,Phase 2.6)。
- 中途離開再回來 → 從第一個未答題續玩(斷點續玩)。
- 重玩:流程相同、可看到比例與新的總結卡,但不再寫入統計;可顯示「你上次選的是這個」。
- 已知限制(接受):清除瀏覽資料、無痕模式、Safari ITP 7 天清除、跨裝置皆會失效。目的為降噪而非防作弊,趣味測驗容忍度高。**不做** IP 記錄或瀏覽器指紋。

### 驗收
- 已有紀錄的題目重答不送 POST;首次作答每題恰送一次。
- 模擬送票失敗後重開站:pending 自動補送、後端只計一次、pending 清空(Phase 2.6)。
- 中途離開再回來,從第一個未答題續玩,已答題不重問。
- 重玩可完整走流程、看比例與新總結卡,統計不再增加;顯示「你上次選的是這個」。

## 9. Phase 規劃

逐 Phase 開工時,才在 `開發設計方針.md` 展開實作契約、在 `測試指南.md` 展開操作清單;本節只列範圍與驗收意圖。

### Phase 1 — 前端答題流程(mock 統計)
- **範圍**:專案骨架(前端選型,輕量為準)、入口頁(§2.1)、題庫首頁(§2.2,OG 除外)、答題頁完整互動(§2.3、§3)、localStorage(§6,「不送 POST」以不呼叫 vote 函式代替)、統計以本地 mock 資料代替 API。含一個開發用示範題庫(佔位圖文)。
- **驗收意圖**:§2.1、§2.3、§3、§6 驗收在 mock 資料下全數成立;手機直式版面正確。

### Phase 2 — 後端(Workers + D1)
- **範圍**:D1 schema 與 migration、GET `/api/stats/:quizId`、POST `/api/vote`、合法清單驗證、種子票 script、前端接真 API(fire-and-forget)。
- **驗收意圖**:§5 驗收全數成立;§2.2「只抓一次統計」成立;答題體感與 Phase 1 無差(零延遲)。

### Phase 2.5 — 公開統計可靠性(**暫緩,條件觸發**)
- **狀態**:不排入開發順序、不阻擋 Phase 3;觸發條件見 §5.4 狀態註記。
- **範圍(觸發時)**:§5.4 全部內容;種子 / 真人分離、Turnstile 匿名 session、後端防重、pending queue、每日彙總。

### Phase 2.6 — 可靠送票
- **範圍**:localStorage `pendingVotes` 與開站 / 恢復連線補送(§5.2、§6);D1 收據表 `(session_id, question_id)` 唯一鍵、收據與計數同交易、冪等 vote;sessionId 為前端隨機匿名 id(§5.4 預案插座)。
- **驗收意圖**:§5、§6 新增驗收成立;暫時失敗的票最終送達且不重複計票;答題體感與 Phase 2 無差。

### Phase 3 — 總結卡 + 分享
- **範圍**:總結卡版面(含**結果主標敘事**:身份標籤四檔、少數派題數、最孤獨的一題,§2.4)與匹配度計算、Web Share(分享文字帶結果摘要)/ 複製連結 fallback、導流卡、per-quiz OG meta tags(含 canonical)、WAE 產品事件(`quiz_start` / `question_answered` / `quiz_completed` / `share_clicked`,fire-and-forget、無 PII)。
- **驗收意圖**:§2.4 驗收全數成立;OG 預覽卡結構正確(真網域實測留待 Phase 5 複驗);四個事件於 Workers Analytics Engine 可查得且不影響答題體感。

### Phase 4 — 題庫內容 ×3
- **範圍**:三個正式題庫(生活、美食、宅宅/影視),每庫一個子階段:題目定稿 → 插圖生成 → 種子票分佈,逐庫與使用者討論。開工時建立 `插圖風格指引.md` 與 `subdocs/題庫/<id>.md`(生活題庫題目已於 2026-07-16 提前定稿,見 `subdocs/題庫/daily-life.md`)。
- **驗收意圖**:§4 圖片規格(WebP、30–50KB、比例與風格一致)全數符合;種子票落在 §5.3 範圍;示範題庫移除或隱藏。

### Phase 5 — 上線
- **範圍**:Pages / Workers / D1 production 部署、網站名稱與網域定案、WAF rate limit、Cloudflare Web Analytics、真機分享實測(LINE / Threads / FB)。
- **驗收意圖**:正式網址完整走完答題與分享一輪;OG 預覽卡實測通過;rate limit 生效且不誤傷正常作答。
