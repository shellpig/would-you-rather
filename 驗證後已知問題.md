## [Phase 2][已修復] stats API 失敗時點選項 TypeError,答題卡死

- **目前處理狀態**:已修復(implementer 2026-07-16 修復並自我驗收通過)。

- **內容說明**:`GET /api/stats/:quizId` 失敗時,`src/lib/statsClient.js > fetchStats` 降級回傳 `{}`;
  之後 `src/pages/quizFlow.js > selectChoice` 呼叫 `computeRatio(snapshot[question.id], choice)` 無防呆,
  `snapshot[question.id]` 為 `undefined` 時丟出 `TypeError`,選項點擊完全失效、答題卡死。
  另一個觸發路徑:題庫 JSON 新增了題目但 `stats` 表尚無該題 row(未跑種子票)時,
  該題在 stats 回應中缺項,同樣命中。

- **修復方式**:兩層防呆,皆照建議解法實作——
  1. `src/lib/statsClient.js > fetchStats`:回應非 2xx 時改為 `throw new Error(...)`,不再降級回傳
     `{}`(fetch 本身的網路錯誤原本就會拋錯,維持不變)。`src/pages/quizFlow.js > startFlow` 用
     `try/catch` 接住,失敗時呼叫新增的 `showStatsError()`,顯示簡單的可重試錯誤畫面
     (「載入失敗,請再試一次」+「重試」按鈕,沿用 `.quiz-intro` 版面與 `.btn-primary` 按鈕樣式,
     新增 `.quiz-intro__error` 文字樣式),不進入題目頁;按「重試」重新呼叫 `startFlow`。
  2. `src/pages/quizFlow.js > selectChoice`:呼叫 `computeRatio` 前先以
     `snapshot[question.id] ?? { a: 0, b: 0 }` 防呆,快照缺項時比例照「快照 + 自己一票」規則計算
     (0+0+自己一票 → 100%/0%)。
  改動範圍收斂在 `statsClient.js` / `quizFlow.js` / `src/styles/main.css`(新增一段錯誤文字樣式),
  未觸碰 Phase 2.6 的 voteQueue / sessionId / vote handler。

- **驗收結果**:375×812 手機視窗,dev server(Vite 5173)+ 本機 Worker(wrangler pages dev
  8788,proxy 轉接)+ 本機 D1 真 API 走查。
  1. 關閉 Worker → 按「開始作答」→ 正確顯示「載入失敗,請再試一次」+「重試」錯誤畫面,不進題目頁,
     console 無 uncaught error(`/api/stats/demo` 請求以 500/`ERR_ABORTED` 失敗,已被
     `try/catch` 接住);重啟 Worker → 按「重試」→ 正常進入第一題。
  2. 模擬單題缺項:暫時從 `src/data/legalList.generated.json`(build 產物)移除 `work-break`
     一題的 id,使 `GET /api/stats/demo` 回應缺該題 key(對應「題庫新增題目但 stats 尚無 row」
     的真實觸發路徑),驗證該題可正常點選兩側、比例正確顯示 100%/0%(含自己一票)、
     「下一題」正常啟用,無 TypeError;其餘題目比例不受影響。驗完以
     `node scripts/generate-legal-list.mjs` 重新產生,還原資料。
  3. 正常路徑回歸:清空 localStorage 後完整走一輪 6 題(首次點選、改選瞬間切換、比例、
     下一題),完成頁與「再玩一次」皆正常;`npm run db:seed` 種子票不受影響。
     （附註:本次自動化瀏覽器環境背景分頁 rAF 被節流,比例條視覺變長動畫的 inline width
     未能觀察到中間值,但這是 Phase 1 已知並驗收過的環境限制,不屬本次改動範圍——
     650ms timeout fallback 使百分比數字仍正確淡入顯示,`下一題」正常啟用。)
  4. `npm test`:38 條全綠(原 36 條 + 新增 `tests/statsClient.test.js` 2 條,涵蓋
     `fetchStats` 失敗拋錯 / 成功回傳兩種情境)。

## [Phase 1][已修復] 首次選擇後比例百分比不顯示

- **目前處理狀態**:已修復(implementer 自我驗收通過)。Phase 1/2 驗收阻擋已解除。

- **內容說明**:首次進入任一題並點選選項後,選中 / 淡化狀態與比例條伸長皆正常,比例數值也已正確
  寫入 DOM,但動畫結束後百分比文字仍維持隱藏(`opacity: 0`)。實測等待超過 0.6 秒 transition 時間後
  仍未出現;只有再次改選另一邊,走「已播過動畫」分支時才會顯示。問題位於
  `src/pages/quizFlow.js > updateRatioSide()`:目前完全依賴 `transitionend` 加上 `.show`,但元素剛從
  `hidden` 切為可見時,單層 `requestAnimationFrame` 未可靠建立 `0% → 目標寬度` 的 transition,
  導致部分瀏覽器不觸發事件。此問題違反 `測試指南.md > Phase 1` 第 7 條,並影響第 11 條及
  `Phase 2` 第 13 條回歸驗收。

- **修復方式**:`src/pages/quizFlow.js > updateRatioSide()`(第一次選擇分支)改用雙層
  `requestAnimationFrame` 可靠提交比例條初始 `width: 0%`(避免元素剛從 `hidden` 轉可見時,單層 rAF
  與「取消 hidden」這次樣式套用被瀏覽器合併成同一次繪製而不觸發 transition),第二層 rAF 才寫入
  目標寬度;同時加入 650ms(略長於 CSS `transition: width 0.6s`)的 `setTimeout` fallback,用 `done`
  旗標防止與 `transitionend` 重複觸發淡入。改選(已播過動畫)分支與其餘行為不變。

- **驗收結果**:375×812 手機視窗,dev server + 本機 Worker/D1 真 API 走查。首次點選（重整頁面、清
  localStorage 後從乾淨狀態)比例條動畫跑完後百分比正確淡入、兩邊加總 100 且為整數;demo 題庫 6 題
  逐題走完,每題首次點選百分比皆正常出現;改選瞬間切換、不重播動畫;續玩、D1 對票、`npm test`
  22 條自動化測試皆不退化。另以「rAF 完全不執行」(比 reduced-motion 更嚴苛的退化情境,實測環境
  背景分頁下 rAF 被瀏覽器節流暫停)驗證 timeout fallback 確實獨立於動畫機制運作,百分比仍正確顯示。
