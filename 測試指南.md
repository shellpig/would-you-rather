# 測試指南

> 測試流程與手動驗收步驟清單。
> **撰寫時機**:各 Phase 的測試項目於**該階段規格定稿時即寫入**(acceptance-test 先行),供實作時據以建測試與自我驗收——不延到驗證時才補。
> **角色歸屬**:verifier-owned。實作者照清單建測試、不得放寬清單,不改本檔(只列建議)。

## 本檔範圍與邊界

- **規格書「驗收」= 意圖層**:什麼狀態必須為真(what must be true),不寫操作步驟。
- **本檔 = 操作層**:開哪個頁、點哪裡、看什麼結果(click-by-click)。
- 同一條斷言不抄兩份;本檔清單項指回規格書對應「驗收」段。
- 規格定稿時,測試項目可能引用尚未定的 content(示範題庫題目等);以中性描述先寫,實作時填實際字樣。

## 測試環境

```
npm install        # 安裝依賴
npm run dev        # 前端開發伺服器(HMR),http://localhost:5173
npm run build      # 產出 dist/(Cloudflare Pages 部署用)
npm run preview    # 本機預覽 build 產物
npm test           # node:test 執行 tests/*.test.js(純函式自動化測試)
npm run db:migrate # 套用本機 D1 migration
npm run db:seed    # 僅供 local / staging 假資料;Phase 2.5 後禁止對 production 執行
```

Worker + D1 開發需兩個終端機:`npm run dev:worker`(8788)與 `npm run dev`(5173,Vite proxy `/api`)。
貼近 production 的單一 origin 驗證:`npm run build`,再執行 `npx wrangler pages dev dist --port 8788`。

- 自動化層:Worker API 與純函式(比例計算、匹配度、localStorage 規則)以自動化測試為主。
- 手動層:互動手感、動畫、手機直式版面、分享行為,依下方清單人工走查(桌機瀏覽器 DevTools 手機模擬 + 真機)。

## 手動驗收清單

### Phase 1 — 前端答題流程(mock 統計)

> 對應 `網站規格書.md` §2.1、§2.3、§3、§6 驗收。示範題庫題目字樣以實作時實際內容為準。

**入口頁(§2.1)**
1. 開 `/`:示範題庫以單欄大卡片顯示,含封面(佔位圖)、標題、題數。
2. mock「玩過人數」設為低於門檻 → 卡片不顯示「N 人玩過」;設為高於門檻 → 顯示。
3. flag off:頁面完全看不到分類 chips;flag on(開發時暫開):chips 可橫向滑動、點選過濾。

**題庫首頁(§2.2,OG 除外)**
4. 點題庫卡 → `/quiz/:slug`:大封面、標題、簡述、「開始作答」鈕、小字「約 1 分鐘 · N 題 · …」。
5. 按「開始作答」→ 進第一題(mock 統計載入,DevTools Network 無 API 請求)。

**答題互動(§3)**
6. 第一題載入:上下兩選項卡皆無比例;進度條與「1/N」正確。
7. 點上選項:上卡進入選中態(主色框、放大),下卡降透明但可點;比例條播變長動畫,`transitionend` 後百分比淡入;兩邊加總 100、整數。
8. 比例含自己一票:與 mock 快照手算核對(所選邊 +1)。
9. 改選下選項:選中態與數值瞬間切換,**動畫不重播**。
10. 按「下一題」→ 第二題;畫面上不存在任何回上一題的途徑;瀏覽器返回鍵不會回到上一題(行為以實作時決策為準,記錄於方針)。
11. 逐題答完,每題動畫各只播一次。

**localStorage 與續玩(§6)**
12. 答到一半關閉分頁重開、重走「開始作答」→ 從第一個未答題續玩,已答題不重問。
13. DevTools 檢查 localStorage:結構為 `{ quizId: { answers: { questionId: choice }, completedAt } }`,記錄與實際選擇一致。
14. 全部答完後重玩:流程可完整走、顯示「你上次選的是這個」;已答題不觸發 vote 送出路徑(Phase 1 驗 mock 層的送票函式不被呼叫,方式實作時定)。

**版面**
15. DevTools 手機直式(375×812 等)走查全流程:上下對決版面、進度條、卡片圖文比例皆正確,無橫向捲動。

### Phase 2.5 — 公開統計可靠性(**暫緩,條件觸發**)

> **狀態(2026-07-16)**:Phase 2.5 整體暫緩(見規格書 §5.4 狀態註記),以下清單僅於啟動時生效,不列入目前覆核範圍。
> 對應 `網站規格書.md` §5.4。驗證目標是公開數字的語意、冪等、暫時故障最終送達與低成本防濫用;
> 不是把匿名趣味測驗證明成「一個自然人一票」。所有 destructive / restore 測試只在獨立 staging D1 執行。

**資料 migration 與統計語意**

1. 對乾淨 D1 套用全部 migration → `stats` 為 `real_a_count / real_b_count`,並建立
   `quiz_sessions`、`session_votes`、`session_events`、`quiz_metrics`、`daily_quiz_metrics`、
   `daily_question_stats`;unique / check / index 與方針一致。
2. production-mode 空庫加入新題庫 → 每題真人 A/B、participant、completion、share 全為 0;
   build / deploy 流程不會自動執行 seed。
3. local / staging seed 可產生假資料,但對 production binding 或 `--remote` 缺少明確確認旗標時必須拒絕;
   假資料不得出現在 production export。
4. 首張有效票後,公開比例的樣本數 = `real_a_count + real_b_count`;首頁「N 人玩過」=
   `quiz_metrics.participant_count`,不再等於第一題票數或任何 seed。

**Turnstile 與匿名 session**

5. 用 Turnstile 官方 test key 建立 session → 後端 Siteverify 成功後才回 token + expiresAt + 完整真人快照;
   `daily_quiz_metrics.start_count` 恰 +1。
6. 缺 token、偽造 token、Siteverify failure、hostname / action 不符、非法 quizId → session 建立被拒,
   D1 不新增 session / start count。
7. 有效 token 只能用於簽章內的 quizId;竄改 payload、過期 token、拿 A 題庫 token 投 B 題庫皆被拒且零寫入。
8. DevTools 走查:同一未過期 session 續玩或 SPA 返回再進入時不重打 Turnstile、不新增 start;
   token 過期時顯示重新驗證,不吞掉 pending。

**投票原子性與冪等**

9. 新 session 對合法題目投 a → `session_votes` 一筆、`stats.real_a_count` +1、當日題目 a +1;
   第一張票另使累積 / 當日 participant 各 +1,全部在成功回應前完成。
10. 同 session / 同題 / 同 choice 連續與並發重送 40 次 → HTTP 皆為 accepted / duplicate success,
    session receipt 仍一筆,累積與每日題目計數總共只 +1。
11. 同 session / 同題改送另一 choice → 409 並回既有 choice;A/B、participant 與每日彙總全部不變。
12. 模擬 transaction 中任一 statement 失敗 → API 不得回成功;receipt、累積、每日計數不可只提交一半。
13. 兩個不同 session 投同一題共 100 個受控 concurrent requests → accepted 數與 D1 增量完全一致;
    duplicate、429、5xx 各自可辨識,沒有無法分類的 silent loss。

**pending queue 與錯誤體驗**

14. 按「下一題」→ 答案與 pending 先寫 localStorage,畫面立即切題;成功 / duplicate response 後才移除 pending。
15. 依序模擬斷網、429 + Retry-After、503 → 作答不卡住、顯示非阻塞「統計同步中」;恢復連線後按
    backoff 補送,最終 pending 清空且 D1 每題只 +1。
16. pending 尚未送達時關閉分頁再開 → app 啟動自動補送;同一題同時最多一個 in-flight request。
17. session / 統計快照 API 斷線或回 4xx / 5xx → 顯示可重試錯誤,不得進入題目頁、不得因空 stats
    發生 uncaught exception。
18. 全部題目已操作但仍有 pending → 完成頁區分「作答完成」與「N 票同步中」;同步完成後狀態更新。

**完成、分享、隱私與營運**

19. 未答完直接呼叫 complete → 被拒;收據齊全後第一次 complete 使累積 / 每日 completion 各 +1,
    重送 40 次仍只 +1。
20. Web Share resolve / clipboard write 成功後才送 share;同 session + method 重送不重複計數;
    cancel / reject 不增加。確認文案沒有把 share_count 稱為曝光或點擊。
21. 查 D1 schema、Worker logs 與 export:不保存 IP、User-Agent、referrer、Turnstile secret、HMAC secret;
    client response / error 亦不洩漏 secret、SQL 或原始 D1 error。
22. staging 套 WAF 規則後走正常 12–16 題流程不被誤傷;burst 超限時收到 429 且 client 保留 pending。
    記錄所用 Cloudflare plan、計數週期、門檻與結果,不得只寫「免費方案每分鐘」。
23. staging 依 runbook 建立 Time Travel bookmark / export,做一次 restore 並核對 stats、receipt、metrics;
    production 只檢查設定與命令,不執行破壞性還原。
24. 記錄 Workers request、D1 read/write、429、5xx 與 pending failure 的觀測位置及升級門檻;
    若缺任一項,不得宣告 P2.5 production-ready。

> 版型沿用 Phase 4 daily-life 清單)。娛樂庫建題定稿時依同版型另列。

**題庫資料**
1. `src/data/quizzes/taiwan-food-wars.json` 15 題的 `question.id`、順序、A / B 文案、圖片路徑與 `subdocs/題庫/taiwan-food-wars.md`「題目定稿表」逐題一致;`titles` 16 組(15 題 + `mainstream`)名稱與判詞與題庫文件「孤獨稱號表」相符。
2. `manifest.json` 含 daily-life 與 taiwan-food-wars 兩庫;美食欄位(cover / category=food / questionCount=15)正確。

**素材規格(規格書 §4)**
3. `public/img/taiwan-food-wars/` 共 31 檔(30 張選項圖 4:3、每張 30–50KB + cover 16:9),`titles/` 下 16 張 512×512 徽章;全為 WebP。

**種子票(規格書 §5.3)**
4. `npm run db:seed` 後查本機 D1:逐題(至少抽查 5 題)與題庫文件「種子票分佈表」定稿值一致,每題兩邊合計約 100;重跑不疊加、不重複 row。

**OG**
5. `npm run build` 後 `dist/quiz/taiwan-food-wars/index.html` 的 og:title / og:description 為題庫文件定稿 OG 文案,og:image / canonical 為絕對網址;`dist/quiz/` 下兩庫靜態頁皆產出。

**整合流程(本機實走)**
6. 入口頁:兩張正式題庫卡皆顯示,達門檻顯示「N 人玩過」;美食佔位卡已移除,娛樂佔位卡仍標「建設中」且不可點。
7. `/quiz/taiwan-food-wars` 題庫首頁:定稿標題 / 簡述 / 封面顯示正確,「已有 N 人作答」與種子票總數相符(達門檻 100)。
8. 15 題逐題作答:圖片全部載入、比例即時顯示且與「種子值 + 自己一票」換算一致(抽查 ≥3 題),兩邊加總 100。
9. 總結卡:身份標籤、孤獨稱號徽章(名稱 + 判詞 + 徽章圖對照題庫文件)、15 題摘要、匹配度皆正確;15 票全數入庫(含最後一題於總結卡情境的補送)。
10. 導流卡「你可能也想玩」雙向驗證:美食總結卡推薦 daily-life、daily-life 總結卡推薦美食,點擊皆正確導向該題庫首頁(同時補驗 Phase 3 #11 的 carry-forward 項)。
11. `npm test` 全綠(不得跳過或放寬既有測試)。
