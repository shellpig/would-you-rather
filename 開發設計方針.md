## Phase 1 — 前端答題流程(實作契約)

### 前端選型

**Vanilla JS + Vite**,不上框架(React / Vue 等皆不用)。理由:
- 整個 Phase 1 只是靜態 SPA + mock 資料,頁面數少(入口頁、題庫首頁、答題頁、完成頁佔位),不需要元件框架的複雜度。
- Vite 提供 dev server(HMR)+ build(bundle、hash 檔名、`public/` 靜態複製)剛好打包成 Cloudflare Pages 能直接部署的 `dist/`,設定成本趨近零。
- 頁面渲染採「模板字串 + innerHTML + 事件委派」手刻,不使用 virtual DOM;規模夠小,不會有可維護性問題。

### Feature flag 機制

`src/config.js` 匯出 `FEATURE_FLAGS = { categoryChips: false }`。`home.js` 依此值決定要不要渲染 `.category-chips`(§2.1 驗收第 3 條)。手動驗收時把值暫改 `true`、重整頁面走查,驗完改回 `false`(勿留在 commit 裡)。

`PLAYED_COUNT_THRESHOLD = 100` 同檔案匯出,`home.js`、`quizFlow.js` 皆引用同一常數搭配 `ratio.js` 的 `shouldShowPlayedCount(count, threshold)` 純函式判定顯示與否。

### Mock 統計來源

`src/data/mockStats.js` 匯出:
- `voteSnapshots`:各題庫、各題的 `{a, b}` 種子票快照,模擬規格書 §5.3(每邊約 20–50 票)。
- `playedCounts`:各題庫「N 人玩過 / 已有 N 人作答」的 mock 數值。

**待確認事項(留給 Phase 2)**:規格書 §4 的 JSON 契約與 §5 的 stats 表只定義了逐題 a/b 計數,並未定義「N 人玩過」這個題庫層級數字的真實來源。Phase 1 暫以獨立 mock 欄位表示;Phase 2 設計後端時需決定實際算法(例如取自題庫第一題的 a+b 總票數,或另建一個計數器 API)。

`src/lib/statsClient.js` 把上述資料包成 `async fetchStats(quizId)` / `async fetchPlayedCount(quizId)`,呼叫端一律 `await`;Phase 2 只需把函式本體換成 `fetch("/api/stats/:quizId")`,呼叫端(`quizFlow.js`、`home.js`)不必改動。「開始作答」時呼叫一次 `fetchStats`,之後全程不再呼叫,滿足規格書 §2.2 驗收。

**驗證兩種門檻情境的方式**:`playedCounts.demo` 預設為高於門檻的值(128),用來展示「顯示」情境;純函式 `shouldShowPlayedCount` 的顯示/隱藏兩個分支已各自寫自動化測試(`tests/ratio.test.js`)覆蓋。若要在畫面上肉眼複驗「低於門檻不顯示」,手動把 `mockStats.js` 的 `playedCounts.demo` 暫改成 <100 的值、重整頁面即可(驗完記得改回 128,勿留在 commit 裡)。

### 瀏覽器返回鍵行為決策

見 `src/router.js` 檔頭註解。摘要:題庫內的「開始作答 → 逐題 → 完成」全程停留在同一個 URL `/quiz/:slug`,題目切換只更新頁面內 JS 狀態,不呼叫 `pushState`/`replaceState`。因此中途按返回鍵會直接離開整個題庫、回到上一個「真正」的 history entry(通常是入口頁 `/`),不會停在題目與題目之間。只有「首頁 → 題庫首頁」這一次導覽會 push 新 entry。已用瀏覽器實測確認(見自我驗收記錄)。

### localStorage 實作決策(§6)

- Key:`wyr:progress`,值為規格書 §6 定義的 `{ [quizId]: { answers, completedAt } }` 結構,一把 key 存全部題庫進度。
- **重玩不覆寫已存答案**:`progressStore.recordAnswer` 對已答過的題目是不動作(no-op),回傳 `isFirstAnswer: false`;`answers` 保存的永遠是「第一次真正送出統計」的選擇,用於顯示「你上次選的是這個」與防止重送 vote。重玩時使用者仍可自由改選,但不影響已存紀錄。
- **`completedAt` 只在第一次完成時寫入**,之後重玩不覆寫(`markCompleted` 對已完成的進度是 no-op),避免重玩洗掉「首次完成時間」這個語意。
- 「你上次選的是這個」提示:單題畫面渲染時,若 `progress.answers[questionId]` 已有值(代表這題在本次以前就答過,只會發生在全部答完後的重玩情境,因為未答完時一律從第一個未答題續玩),在對應選項卡下顯示提示文字。

### 完成頁範圍說明

Phase 1 不含規格書 §2.4 的正式總結卡(比對比例、匹配度、Web Share/複製連結、導流卡屬 Phase 3 範圍)。目前 `quizFlow.js` 的 `showComplete()` 只是一個佔位頁:標題「測驗完成!」、逐題你選了什麼的條列、「再玩一次」/「回首頁」兩個按鈕。用途是讓 §6 的 `completedAt` 寫入邏輯與重玩流程可以被驗收,不是最終 UI。

### 本機指令

```
npm install       # 安裝依賴(vite)
npm run dev       # 開發伺服器(HMR),預設 http://localhost:5173
npm run build     # 產出 dist/(含 public/ 靜態檔複製、_redirects)
npm run preview   # 本機預覽 build 產物
npm test          # node:test 執行 tests/*.test.js(純函式自動化測試)
```

`npm run dev` / `npm run preview` 皆會自動以 SPA fallback 處理 `/quiz/:slug` 這類非根路徑的直接訪問或重整(Vite 預設 `appType: 'spa'`);正式部署則靠 `public/_redirects` 讓 Cloudflare Pages 做同樣的事。
