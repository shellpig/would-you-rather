## Phase 1 — 前端答題流程(實作契約)

### 前端選型

**Vanilla JS + Vite**,不上框架(React / Vue 等皆不用)。理由:
- 整個 Phase 1 只是靜態 SPA + mock 資料,頁面數少(入口頁、題庫首頁、答題頁、完成頁佔位),不需要元件框架的複雜度。
- Vite 提供 dev server(HMR)+ build(bundle、hash 檔名、`public/` 靜態複製)剛好打包成 Cloudflare Pages 能直接部署的 `dist/`,設定成本趨近零。
- 頁面渲染採「模板字串 + innerHTML + 事件委派」手刻,不使用 virtual DOM;規模夠小,不會有可維護性問題。

### Feature flag 機制

`src/config.js` 匯出 `FEATURE_FLAGS = { categoryChips: false }`。`home.js` 依此值決定要不要渲染 `.category-chips`(§2.1 驗收第 3 條)。手動驗收時把值暫改 `true`、重整頁面走查,驗完改回 `false`(勿留在 commit 裡)。

### 瀏覽器返回鍵行為決策

見 `src/router.js` 檔頭註解。摘要:題庫內的「開始作答 → 逐題 → 完成」全程停留在同一個 URL `/quiz/:slug`,題目切換只更新頁面內 JS 狀態,不呼叫 `pushState`/`replaceState`。因此中途按返回鍵會直接離開整個題庫、回到上一個「真正」的 history entry(通常是入口頁 `/`),不會停在題目與題目之間。只有「首頁 → 題庫首頁」這一次導覽會 push 新 entry。已用瀏覽器實測確認(見自我驗收記錄)。

### localStorage 實作決策(§6)

- Key:`wyr:progress`,值為規格書 §6 定義的 `{ [quizId]: { answers, completedAt } }` 結構,一把 key 存全部題庫進度。
- **重玩不覆寫已存答案**:`progressStore.recordAnswer` 對已答過的題目是不動作(no-op),回傳 `isFirstAnswer: false`;`answers` 保存的永遠是「第一次真正送出統計」的選擇,用於顯示「你上次選的是這個」與防止重送 vote。重玩時使用者仍可自由改選,但不影響已存紀錄。
- **`completedAt` 只在第一次完成時寫入**,之後重玩不覆寫(`markCompleted` 對已完成的進度是 no-op),避免重玩洗掉「首次完成時間」這個語意。
- 「你上次選的是這個」提示:單題畫面渲染時,若 `progress.answers[questionId]` 已有值(代表這題在本次以前就答過,只會發生在全部答完後的重玩情境,因為未答完時一律從第一個未答題續玩),在對應選項卡下顯示提示文字。

### 完成頁範圍說明

Phase 1 不含規格書 §2.4 的正式總結卡(比對比例、匹配度、Web Share/複製連結、導流卡屬 Phase 3 範圍)。目前 `quizFlow.js` 的 `showComplete()` 只是一個佔位頁:標題「測驗完成!」、逐題你選了什麼的條列、「再玩一次」/「回首頁」兩個按鈕。用途是讓 §6 的 `completedAt` 寫入邏輯與重玩流程可以被驗收,不是最終 UI。

## Phase 2 — 後端 Workers + D1(實作契約)

### 後端載體:Cloudflare Pages Functions(不是獨立 Worker)

規格書 §1 寫「Cloudflare Workers」,但 Pages Functions 本質上就是跑在同一個 Workers runtime 上的
Worker(Cloudflare 官方文件語:"Functions are just Workers"),差別只在部署與路由慣例。選擇
Pages Functions 而非獨立 Worker 專案,理由:

- **同 repo、同一次部署**:前端(Phase 1 的 Vite 靜態站,產出到 `dist/`)與 API 用同一個
  `wrangler.toml`(`pages_build_output_dir = "dist"`)、同一次 `wrangler pages deploy` 上線,
  Phase 5 不必維護兩個獨立專案與兩份設定。
- **同源、免 CORS**:Pages Functions 與靜態資產同一個 origin 服務,前端 `fetch("/api/...")`
  直接打相對路徑即可,不需處理跨源標頭。
- **本機開發更單純**:`wrangler pages dev` 一個指令就能同時模擬靜態站 + Functions + D1
  綁定,不必另外起一個獨立 Worker dev server 再手動接 D1 binding。
- 檔案結構走 Cloudflare Pages Functions 慣例:`/functions/api/**` 內的檔案依檔名/資料夾自動路由
  (`functions/api/vote.js` → `/api/vote`;`functions/api/stats/[quizId].js` → `/api/stats/:quizId`),
  底線開頭的目錄(`_lib/`)不會被當成路由,用來放共用程式碼。

若日後需要 API 獨立擴充/獨立部署(例如流量大到要跟靜態站分開 scale),可再拆成獨立 Worker,
屆時把 `functions/api/**` 的程式碼原封搬過去即可,邏輯不綁定 Pages 專屬 API。

### wrangler 設定與本機開發方式

`wrangler.toml`(專案根目錄):
```toml
name = "would-you-rather"
compatibility_date = "2026-07-16"
pages_build_output_dir = "dist"

[[d1_databases]]
binding = "DB"
database_name = "would-you-rather-db"
database_id = "<本機佔位值,見檔內註解>"
```
`database_id` 在本機 `--local` 模式下只用來命名 `.wrangler/state` 內的 SQLite 檔,不會連線
Cloudflare 帳號驗證,可以先用隨機 UUID 佔位。**Phase 5 正式部署前**需執行
`wrangler d1 create would-you-rather-db` 取得真正的 database_id 並替換此值,否則
`--remote` / 正式部署會失敗。

本機開發分兩種情境,擇一使用:

1. **只改前端(不需要真 API)**:`npm run dev`(Vite,5173,行為與 Phase 1 相同)。
2. **前端 + API 一起跑(Phase 2 起的預設開發方式)**:兩個終端機視窗:
   - 終端機 A:`npm run dev:worker`(`wrangler pages dev public --port 8788`)——只用
     `public/` 目錄起靜態伺服(內容不影響,實際頁面走終端機 B 的 Vite),重點是掛載
     Functions + 本機 D1 綁定於 8788 埠。
   - 終端機 B:`npm run dev`(Vite,5173,HMR)。`vite.config.js` 設定 `server.proxy`
     把 `/api/*` 轉給 `http://localhost:8788`,瀏覽器只需開 `http://localhost:5173`,
     `fetch("/api/...")` 對前端程式碼而言仍是同源相對路徑,不必處理跨埠 CORS。

   選這個雙終端機方案而非 `wrangler pages dev -- <command>` 的 proxy 子指令:該子指令語法
   在目前 wrangler 版本(4.111)已標示 `[deprecated]`(見 `wrangler pages dev --help`),用
   Vite 官方的 `server.proxy` 是更穩定、不依賴即將移除功能的做法。

生產環境驗證(貼近正式部署行為,單一 origin,無 proxy):
```
npm run build              # 產出 dist/
npx wrangler pages dev dist --port 8788
```
`wrangler.toml` 的 `[[d1_databases]]` 綁定與 `pages_build_output_dir` 皆會自動套用,不需額外
`--d1` 旗標。

### D1 migration

`migrations/0001_init.sql`,schema 照規格書 §5.1 一字不差:
```sql
CREATE TABLE stats (
  quiz_id     TEXT,
  question_id TEXT,
  a_count     INTEGER DEFAULT 0,
  b_count     INTEGER DEFAULT 0,
  PRIMARY KEY (quiz_id, question_id)
);
```
本機套用:`npm run db:migrate`(`wrangler d1 migrations apply would-you-rather-db --local`)。

### 合法清單:build 時從題庫 JSON 產生

`scripts/generate-legal-list.mjs` 讀 `src/data/manifest.json` + `src/data/quizzes/*.json`,
對每個題庫輸出 `{ firstQuestionId, questionIds }`,寫成 `src/data/legalList.generated.json`
(已列入 `.gitignore`,屬 build 產物,不進 repo)。選擇「build 時產生一個靜態 JSON」而非
「Worker 執行期動態讀檔」或「逐檔 dynamic import」,理由:

- Cloudflare Workers runtime 沒有檔案系統,`fs.readdir` 這類在執行期掃描 `quizzes/*.json`
  的做法行不通;而動態組出 per-id 的 `import()` 路徑,esbuild 無法靜態分析、無法可靠 bundle。
- 產生單一靜態 JSON 後,`functions/api/_lib/legalList.js` 只需一行 `import ... from
  "../../../src/data/legalList.generated.json"`,wrangler 在 bundle Functions 時原生支援
  `.json` 的 esbuild loader,一併打包進 Worker,執行期零額外 I/O、零檔案系統依賴。

執行時機:`predev` / `prebuild` npm hook 自動跑(`npm run dev`、`npm run build` 前都會重新產生),
不需手動呼叫;`scripts/seed-votes.mjs` 內部也會先重跑一次確保清單與當下題庫 JSON 同步。

`functions/api/_lib/legalList.js` 匯出 `isLegalQuiz`、`isLegalQuestion`、`getQuestionIds`、
`getFirstQuestionId`、`getAllQuizIds`,`functions/api/vote.js`、`functions/api/stats/[quizId].js`、
`functions/api/played-counts.js` 皆靠這層驗證,拒絕非法 quizId/questionId/choice
(規格書 §5.2)。

### 種子票 script

`scripts/seed-votes.mjs`(`npm run db:seed`):對合法清單內每一題,產生 20–50 的隨機
`a_count`/`b_count`,組成多筆 `INSERT INTO stats (...) VALUES (...) ON CONFLICT(quiz_id,
question_id) DO UPDATE SET a_count = ..., b_count = ...` 寫進暫存 `.sql` 檔,再用
`wrangler d1 execute would-you-rather-db --local --file=<tmp>` 一次性套用,執行後刪除暫存檔。

**可安全重跑**的方式:用 `ON CONFLICT ... DO UPDATE`(不是 `INSERT` 或 `INSERT OR IGNORE`),
重跑不會產生重複 row、不會報 PRIMARY KEY 衝突,只會把該題兩邊計數換成新的一組隨機值——
不是「數值不變」意義上的冪等,而是「結構安全、可重複執行、不累加爆量」。加 `--remote`
旗標可對正式 D1 執行(Phase 5 上線後使用,平時不要對正式庫重跑,會洗掉真實票數)。

### 「N 人玩過」資料來源決策

**決策**:新增 `GET /api/played-counts`,一次回傳所有合法題庫的
`{ quizId: count }`,`count` = 該題庫**第一題**的 `a_count + b_count`。

**理由**:
- 答題流程單向、無回上一題(規格書 §8),第一題是唯一「保證每個開始作答的人都會投的
  一題」,用它的票數總和當「玩過人數」的下限估計,語意上最貼近「N 人玩過」而不需要
  另建一張計數器表或在 `stats` 之外多維護一個 counter row。
- **取捨(已知會灌水)**:種子票（§5.3,每題各自 20–50 隨機)也會讓第一題的種子值計入
  「玩過人數」,新題庫上線初期這個數字不是「真實玩過人次」而是「種子基準 + 真實票數」。
  這與種子票本身「避免 100% vs 0% 尷尬」的取捨精神一致(規格書 §5.3 本來就接受早期
  數字非真實分佈),記錄於此供 Phase 4/5 建題與檢視數字時知悉;若未來要拆分「種子墊底」
  與「真實玩過人次」,需另外加欄位或表,目前判斷不值得為此增加 schema 複雜度。
- **入口頁多題庫 / API 面積最小**:入口頁(`home.js`)對 manifest 內每個題庫各呼叫一次
  `fetchPlayedCount(quizId)`(呼叫端沿用 Phase 1 寫法,不因後端改動而改)。若讓
  `fetchPlayedCount` 直接一對一打 API,題庫一多就會是 N 個請求。改為
  `GET /api/played-counts` 一次回傳全部題庫的值,`src/lib/statsClient.js` 內用一個
  模組層級的快取 `Promise` 把多次呼叫合併成同一個 HTTP 請求(見該檔案 `loadPlayedCounts`),
  呼叫端介面完全不變、API 面積維持在 3 支端點(`GET /api/stats/:quizId`、
  `POST /api/vote`、`GET /api/played-counts`)。
- 題庫首頁(`quizFlow.js`)的 `fetchPlayedCount(slug)` 同樣吃這個快取,不會因為「開始作答」
  前需要顯示「已有 N 人作答」而額外打一次 `GET /api/stats/:quizId`(那支只在按下「開始
  作答」當下呼叫一次,維持 §2.2 驗收)。

### API 契約

**`GET /api/stats/:quizId`**
- 200:`{ [questionId]: { a: number, b: number } }`,涵蓋該題庫合法清單內所有題目;
  尚未有票的題目回傳 `{a:0,b:0}`(不因 D1 缺 row 而漏欄位)。
- 404(非規格書明文要求,實作補充):`quizId` 不在合法清單 → `{ "error": "unknown quizId" }`。

**`GET /api/played-counts`**
- 200:`{ [quizId]: number }`,涵蓋合法清單內所有題庫。

**`POST /api/vote`**
- body:`{ quizId, questionId, choice: "a"|"b" }`。
- 200:`{ "ok": true }`,並以單一 SQL statement
  `INSERT ... ON CONFLICT(quiz_id, question_id) DO UPDATE SET a_count = a_count + 1`
  (或 b_count)原子遞增——SQLite 保證單一 statement 的原子性,並發寫入不掉票。
- 400:body 缺欄位/型別錯誤、`choice` 不是 `"a"`/`"b"`,或 `quizId`/`questionId` 不在
  合法清單 → `{ "error": "..." }`,不寫入 D1。

## Phase 2.5 — 公開統計可靠性(實作契約)——**暫緩,條件觸發**

> **狀態(2026-07-16 站方決策)**:本段整體為預案,未觸發前不實作;觸發條件見 `網站規格書.md > §5.4` 狀態註記。
> Phase 2 現況(種子票計入統計 + fire-and-forget + localStorage 降噪)即為完成態,不阻擋 Phase 3。

> 本 Phase 在 Phase 2 基礎 API 與 Phase 3 UI 之間插入。目標不是建立帳號或追蹤個人,而是讓
> production 的公開比例具備明確語意、可重試且不易被低成本腳本污染。若本節與 Phase 2 的
> seed / fire-and-forget 決策衝突,以本節為正式上線契約。

### 核心取捨

- 保留 Pages Functions + D1,不為 MVP 引入另一套交易資料庫。
- production 不使用合成種子票。公開比例只算後端接受的真人票,並顯示樣本數;即使早期出現
  100% / 0%,也優先保持數據語意誠實。
- 不追求「一個自然人只能投一次」。Turnstile + 匿名 session + WAF 只提高自動灌票成本並防止
  同一 session 重送;清除儲存、換裝置或人工重做仍可能新增樣本,畫面不得包裝成科學民調。
- 使用者操作保持零等待,但「不阻塞 UI」不再等於「失敗即丟棄」;改用 client pending queue +
  server idempotency 做最終送達。

### D1 schema 演進

Phase 2 尚未正式部署,Phase 2.5 migration 可重建本機 `stats` 並清除 demo seed。production 初次建立
直接使用下列最終 schema,不要把 Phase 2 的隨機值搬入 `real_*` 欄位。

```sql
CREATE TABLE stats (
  quiz_id      TEXT,
  question_id  TEXT,
  real_a_count INTEGER NOT NULL DEFAULT 0,
  real_b_count INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (quiz_id, question_id)
);

CREATE TABLE quiz_sessions (
  session_id     TEXT PRIMARY KEY,
  quiz_id        TEXT NOT NULL,
  created_at     INTEGER NOT NULL,
  first_vote_at  INTEGER,
  completed_at   INTEGER,
  expires_at     INTEGER NOT NULL
);

CREATE INDEX quiz_sessions_quiz_id ON quiz_sessions (quiz_id);

CREATE TABLE session_votes (
  session_id   TEXT NOT NULL,
  question_id TEXT NOT NULL,
  choice       TEXT NOT NULL CHECK (choice IN ('a', 'b')),
  accepted_at  INTEGER NOT NULL,
  counted_at   INTEGER,
  PRIMARY KEY (session_id, question_id),
  FOREIGN KEY (session_id) REFERENCES quiz_sessions(session_id)
);

CREATE TABLE session_events (
  session_id TEXT NOT NULL,
  event_type TEXT NOT NULL CHECK (event_type IN ('web-share', 'copy-link')),
  accepted_at INTEGER NOT NULL,
  PRIMARY KEY (session_id, event_type),
  FOREIGN KEY (session_id) REFERENCES quiz_sessions(session_id)
);

CREATE TABLE quiz_metrics (
  quiz_id           TEXT PRIMARY KEY,
  participant_count INTEGER NOT NULL DEFAULT 0,
  completion_count  INTEGER NOT NULL DEFAULT 0,
  share_count       INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE daily_quiz_metrics (
  day               TEXT,
  quiz_id           TEXT,
  start_count       INTEGER NOT NULL DEFAULT 0,
  participant_count INTEGER NOT NULL DEFAULT 0,
  completion_count  INTEGER NOT NULL DEFAULT 0,
  share_count       INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (day, quiz_id)
);

CREATE TABLE daily_question_stats (
  day          TEXT,
  quiz_id      TEXT,
  question_id  TEXT,
  a_count      INTEGER NOT NULL DEFAULT 0,
  b_count      INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (day, quiz_id, question_id)
);
```

時間欄位統一用 Unix milliseconds UTC;`day` 統一為 UTC `YYYY-MM-DD`,避免部署區域與日光節約時間
造成跨日不一致。`session_id` 用 cryptographically secure random 產生;D1 只存 id,不存 IP、
User-Agent、referrer 或指紋。referrer / page view 交給 Cloudflare Web Analytics。

`session_votes` 是短期防重收據,不是長期分析事件表。token 到期後的清理期限與排程在 Phase 5
runbook 定稿;刪除前先確認累積與每日彙總均已完成。長期報表只讀 `stats`、`quiz_metrics` 與
兩張 daily 表。

### Turnstile 與匿名 session

新增 `POST /api/session`:

```json
{ "quizId": "daily-life", "turnstileToken": "..." }
```

處理順序:

1. 驗證 request method、JSON content-type、body 大小與合法 quizId。
2. 後端以 secret 呼叫 Turnstile Siteverify;檢查 `success`、hostname 與 action。只信任 Siteverify
   回應,不信任前端自行宣告成功。
3. 產生 `session_id`,寫入 `quiz_sessions`,並把 `daily_quiz_metrics.start_count` +1。
4. 以 server secret 做 HMAC 簽章,回傳題庫限定的 opaque `sessionToken`、`expiresAt` 與當下
   真人統計快照。Turnstile secret / HMAC secret 只放 production secret binding,不進 repo。

成功回應:

```json
{
  "sessionToken": "<opaque>",
  "expiresAt": 0,
  "stats": { "coffee-tea": { "a": 0, "b": 0 } }
}
```

將統計快照併入 session 回應,取代 Phase 2「開始時另打一支 GET stats」,所以開始作答仍只有一次
後端 round trip。既有 `GET /api/stats/:quizId` 可保留供公開讀取 / 除錯,但正式答題流程不再靠它
建立 session。

client 將每題庫的 token 放在既有 `wyr:progress`。未過期 token 可續玩,不得為 SPA 內返回首頁再進入
而重複建立 session。token 過期且仍有未同步票時,顯示重新驗證流程;不得擅自丟棄 pending。

### 冪等投票交易

`POST /api/vote` body 改為:

```json
{
  "sessionToken": "<opaque>",
  "questionId": "coffee-tea",
  "choice": "a"
}
```

後端先驗 token 簽章、期限、quizId 與合法題目。接受新票時,在同一 `env.DB.batch()` 交易邊界內:

1. `INSERT OR IGNORE session_votes`。
2. 只對 `counted_at IS NULL` 的該收據遞增 `stats.real_*` 與 `daily_question_stats`。
3. 若這是 session 第一張有效票,設定 `first_vote_at`,並將累積 / 每日 participant 各 +1。
4. 設定收據 `counted_at`,最後才回成功。

D1 batch 內每個 statement 必須用條件確保只有尚未 counted 的收據能加總;不能採「先查有沒有 → 再寫」
的非原子 read-modify-write。回應語意:

- 新票完成全部寫入:`200 { "ok": true, "duplicate": false }`。
- 同 session / 同題 / 同 choice 重送:`200 { "ok": true, "duplicate": true }`,所有計數不變。
- 同 session / 同題但 choice 不同:`409`,回傳已接受的 choice,所有計數不變。
- 無效 / 過期 token:`401`;payload 或題目非法:`400`;過載或暫時性 D1 問題:`503` + `Retry-After`。

不要把 D1 原始錯誤、SQL 或 secret 回給 client。成功回應只允許在收據與所有彙總同時提交後送出。

### 完成與分享事件

新增兩個冪等端點:

- `POST /api/complete { sessionToken }`:後端確認該 session 對題庫所有合法題目都有收據後才設定
  `completed_at`,累積 / 每日 completion 各 +1;重送回 duplicate success。
- `POST /api/share { sessionToken, method: "web-share"|"copy-link" }`:只在前端 Web Share promise resolve
  或 clipboard write 成功後呼叫;以 `session_id + method` 的收據防止同一完成頁重試灌高。此數字只代表
  成功觸發分享操作,不代表貼文曝光或他人點擊。

`participant_count` 取代 Phase 2「第一題 a+b」的玩過人數推估。首頁 `GET /api/played-counts`
改讀 `quiz_metrics.participant_count`,不再受題目排序、種子票或第一題重複請求影響。
client 只在所有 pending vote 都收到確認後呼叫 complete;若仍有缺少收據,complete 回可重試的 conflict,
完成事件留在本機 pending,待票數同步後再送。

### client pending queue

localStorage 每題庫擴充:

```json
{
  "answers": { "coffee-tea": "a" },
  "sessionToken": "<opaque>",
  "pendingVotes": {
    "coffee-tea": { "choice": "a", "queuedAt": 0, "attempts": 0 }
  },
  "completedAt": null
}
```

按「下一題」時先原子地把答案與 pending 項目寫入 localStorage,再切題並背景呼叫 vote。只有收到
`200 accepted / duplicate` 才刪 pending;`409` 以後端既有 choice 校正本機狀態;網路錯誤、429、503
保留 pending,採有上限的 exponential backoff + jitter。在以下時機 flush:

- app 啟動 / 題庫續玩;
- browser `online` event;
- 本次作答下一個正常網路空檔;
- 完成頁載入。

同一題同時最多一個 in-flight request。答題 UI 不等待 vote,但有 pending 時顯示非阻塞「統計同步中」;
完成頁需區分「作答完成」與「N 票尚在同步」。`fetchStats` / session 建立失敗時顯示重試畫面,
不可回 `{}` 後繼續呼叫 `computeRatio(undefined, ...)`。

### 低樣本呈現

- 比例使用 `real_a_count / real_b_count`,選擇後可在畫面上先加自己的 pending 票做即時預覽。
- 比例旁顯示「目前 N 票」;N 小也照實顯示,不以 seed 修飾。
- 總結卡匹配度以開始時的真人快照 + 本輪選擇計算,並標示為趣味參考;不得稱民調或代表人口。
- production seed script 必須停用 / 加明確 guard。允許 local / staging 假資料,但資料庫 binding 與
  production 完全分離。

### 防濫用、容量與復原

- Turnstile 保護 session issuance;WAF 的唯一免費 rate-limit rule 優先涵蓋 session / vote 寫入路徑。
  免費方案可用欄位、計數週期與規則數有限,Phase 5 依當時官方方案與遠端壓測決定門檻,文件不得再
  寫死「同 IP 每分鐘」。
- rate limit 只負責減速;token + D1 unique constraint 才負責同 session 防重。CORS / Referer 不作安全邊界。
- staging 做 100 concurrent vote burst:所有 request 分類為 accepted / duplicate / 429 / retryable 5xx;
  accepted 數須與 D1 增量完全相等,client 重試後最終無 pending、無重複票。正式流量目標確定後再提高壓測量。
- 監看 Workers request、D1 rows read/write、429、5xx、pending flush failure。接近 Free 限額或推廣前
  直接升 Workers Paid,不要靠漏票維持免費額度。
- Phase 5 runbook 記錄 D1 Time Travel bookmark / restore、production export、migration rollback 與
  seed guard 驗證。恢復演練必須在 staging 先做,不得第一次就在 production 嘗試。

## Phase 2.6 — 可靠送票(實作契約)

> 範圍見 `網站規格書.md > §9 Phase 2.6`、§5.2、§6。目標:送票在斷網 / Worker 暫時掛掉時最終送達,
> 且重送不重複計票;答題體感與 Phase 2 完全無差(零等待)。

### sessionId:per-quiz、前端產生、存於該題庫的 progress 物件

- **範圍決策**:sessionId **不是**全站共用一個 id,而是每個題庫各自一組,產生後存進該題庫的
  localStorage progress 物件(`src/lib/sessionId.js` 產生、`src/lib/progressStore.js` 的
  `ensureSessionId` 寫入,見下方 `client pendingVotes` 段的資料結構)。
- **理由(與 §5.4 插座相容)**:規格書 §5.2 / §9 literal 寫的收據表唯一鍵是
  `(session_id, question_id)`,沒有併入 `quiz_id`。question_id 本身只在單一題庫內唯一(`stats`
  表的 primary key 是 `(quiz_id, question_id)`,可見不同題庫可能有同名 question_id)。若
  sessionId 是全站共用一個,`(session_id, question_id)` 就不足以避免跨題庫同名 question_id 互相
  覆蓋。讓 sessionId **per-quiz** 產生,可以讓 `(session_id, question_id)` 天生等價於
  `(session_id, quiz_id, question_id)`——不需要動規格書 literal 寫的唯一鍵字面,問題就已經不存在。
  這個設計也剛好與 §5.4(暫緩預案)`quiz_sessions` 的 `quiz_id` 欄位精神一致(該表本來就是
  「題庫限定的匿名 session」),Phase 2.6 → 2.5 之間不需要遷移語意。
- **產生方式**:`crypto.randomUUID()`,不支援時 fallback 手動組 UUID v4 格式字串(見
  `src/lib/sessionId.js`)。固定輸出 UUID v4 格式,讓後端可以用簡單正則驗證格式合法性
  (`functions/api/vote.js` 的 `SESSION_ID_PATTERN`)。
- **產生時機**:該題庫第一次真正需要送票時(`quizFlow.js` 的 `goNext`,`isFirstAnswer === true`
  的當下)才產生,不在「開始作答」或頁面載入時就先建立——沒答過題就不需要 sessionId。
- 現階段後端**不驗證**sessionId 的身份,只驗證格式;這正是 §5.4 的插座——之後啟動 Turnstile 時,
  只需要在「發 session」的步驟加上 Siteverify 驗證閘,收據表與這裡的冪等邏輯不必更動。

### vote handler:收據插入與計數遞增同一交易

`functions/api/vote.js` body 增加 `sessionId`(格式用正則驗證,不驗身份)。核心設計:用
**D1 `batch()` + SQLite `changes()`** 讓「收據插入」與「計數是否要加」在同一個交易內原子完成,
不需要「先查有沒有 → 再寫」的 non-atomic read-modify-write:

```sql
-- statement 1
INSERT INTO vote_receipts (session_id, quiz_id, question_id, choice, created_at)
VALUES (?, ?, ?, ?, ?)
ON CONFLICT (session_id, question_id) DO NOTHING;

-- statement 2(aDelta/bDelta 由呼叫端依 choice 算好,值是 0 或 1)
INSERT INTO stats (quiz_id, question_id, a_count, b_count)
VALUES (?, ?, ? * changes(), ? * changes())
ON CONFLICT (quiz_id, question_id) DO UPDATE SET
  a_count = a_count + ? * changes(),
  b_count = b_count + ? * changes();
```

- D1 batch 內的多個 statement 在同一個連線循序執行於同一個交易,SQLite 的 `changes()` 函式
  會延續上一個 statement 影響的 row 數:新收據(INSERT 成功)為 1,重複送達(唯一鍵衝突,
  `DO NOTHING` 生效)為 0。statement 2 把這個 0/1 當乘數,決定 stats 要不要真的 +1。
- 這個寫法在實作前先用 `wrangler d1 execute --command` 手動驗證過 `changes()` 跨 statement
  傳遞的行為,實作後再用 `tests/api.test.js` 的並發測試(10 個相同 session+question 同時打)
  驗證過在真實 `wrangler pages dev`(workerd/D1 本機模擬)下並發也不會多算——結果符合預期。
- 回應語意(Phase 2.6 版本,比 §5.4 預案簡單):無論新票或重複送達,一律 `200 { ok: true }`;
  重複送達(不論 choice 是否相同)不加票、不改寫 `vote_receipts.choice`(`DO NOTHING` 不會更新
  既有 row)。§5.4 預案的 `duplicate` 欄位與 `409` 衝突語意留給日後真正需要區分時再加。
- 缺 `sessionId` 或格式不合法(非 UUID 格式)→ 400,不寫入 D1;`quizId`/`questionId`/`choice`
  驗證邏輯與 Phase 2 相同(合法清單 + enum)。

### client pendingVotes:結構、寫入時機、補送

localStorage 每個題庫的 progress 物件擴充(`src/lib/progressStore.js`):

```json
{
  "answers": { "morning-drink": "a" },
  "sessionId": "24f60304-525e-4915-aa6b-2a3b3585b559",
  "pendingVotes": { "wine": { "choice": "a", "queuedAt": 1752600000000 } },
  "completedAt": null
}
```

- **寫入時機**(`src/pages/quizFlow.js` 的 `goNext`):只在 `isFirstAnswer === true`(規格書 §6
  「已答不重送」既有邏輯,補送本身不算新答題)時才 `ensureSessionId` + `queuePendingVote`,
  answers / sessionId / pendingVotes 三者一起原子寫入 localStorage(單一
  `saveQuizProgress` 呼叫),接著才呼叫 `sendVote(...)` 背景送出、**不 await**——切題邏輯
  緊接著同步執行,不等任何網路回應,答題體感與 Phase 2 無差。
- **清除時機**(`src/lib/voteQueue.js` 的 `sendVote`):送出後掛 `.then()`,只有 `res.ok`
  (2xx)才讀最新 progress、`clearPendingVote`、寫回 localStorage;失敗(斷網 reject 或非 2xx)
  一律保留 pending,交給下次補送時機重試,不影響答題流程(規格書 §5 驗收)。
- **補送時機**(`src/lib/voteQueue.js` 的 `flushPendingVotes`,由 `src/main.js` 在 app 啟動時
  呼叫一次、並註冊 `window.addEventListener("online", ...)`):掃描**所有題庫**(不只目前開著
  的這個)的 pendingVotes,逐筆呼叫 `sendVote`。
- **7 天丟棄**(`progressStore.js` 的 `prunePendingVotes`):`queuedAt` 超過 7 天的項目在補送
  掃描時直接移除、**不發請求**,丟棄結果立即寫回 localStorage。
- **同題單一 in-flight**(`voteQueue.js` 的 `inFlight` Set,key 為 `` `${quizId}::${questionId}` ``):
  補送掃描與「下一題」觸發的即時送出共用同一個 `sendVote`,避免同一題同時有兩個請求在飛
  (即使唯一鍵最終也會擋掉多餘的計數,這裡是額外省一次不必要的網路請求)。
- **舊資料相容**:Phase 2.6 之前存的 progress 物件沒有 `sessionId`/`pendingVotes` 欄位,
  `storage.js` 的 `getQuizProgress` 讀取時用 `{ ...createEmptyQuizProgress(), ...stored }`
  補上預設值,不影響既有的 `answers`/`completedAt`。

### 實作中發現並修正的一個 bug(quizFlow.js 記憶體內 progress 與 localStorage 分歧)

手動驗收第 3 條(關 Worker 答題 → 補送)過程中發現:`quizFlow.js` 的 `goNext` 在多題之間
共用同一個閉包變數 `progress`;但 `voteQueue.js` 的 `sendVote` 成功回呼是直接對 localStorage
做「讀最新 → 清 pending → 寫回」,不會同步更新 `quizFlow.js` 手上那份記憶體內的 `progress`。
如果送票成功發生在使用者答下一題**之前**,下一次 `goNext` 用舊的記憶體內 `progress`(仍帶著
已經送達的 pending 項目)整包存回 localStorage,會把剛清除的 pending 又救回來——不會多算票
(D1 冪等擋得住),但 pending 永遠不會真正清空,會一直重送已經成功的票。修法:`goNext` 在
合併新的 pending 項目前,先用 `getQuizProgress(slug).pendingVotes` 同步一次最新狀態,只覆寫
`pendingVotes` 這個唯一會被外部非同步寫入的欄位,其餘欄位(answers/sessionId/completedAt)
沿用記憶體內的值(這兩者只有 `quizFlow.js` 自己會寫,不會被外部競爭)。已用瀏覽器手動測試
（關 Worker 答兩題、重啟 Worker、確認只補送那兩筆且不含已送達的第一題)驗證修復有效。

### 已知問題(pre-existing,Phase 2 遺留)——已於 2026-07-16 修復

手動驗收過程中另外發現:若 `GET /api/stats/:quizId`(開始作答時抓的統計快照)失敗,
`src/lib/statsClient.js` 的 `fetchStats` 會降級回傳 `{}`(非拋錯),但 `quizFlow.js` 的

**修復**(2026-07-16,詳細修復方式與驗收結果見 `subdocs/操作說明/驗證後已知問題.md`):`fetchStats` 改為失敗時

`selectChoice` 對單題缺項以 `snapshot[question.id] ?? {a:0,b:0}` 防呆。

### 資料推導:`src/lib/resultSummary.js`(純函式)

輸入固定用 `quiz`(題庫定義)、`progress.answers`(規格書 §6「第一次真正送出統計的選擇」)、
`snapshot`(`quizFlow.js` 開始作答時抓的統計快照,與答題頁 `selectChoice` 同一份)——
確保「主標區所有比例與答題頁口徑一致」(規格書 §2.4 驗收)。

**決策(總結卡用「本輪選擇」覆蓋 `answers`;2026-07-17 推翻原決策)**:`showComplete()`
以 `{ ...progress.answers, ...currentChoices }` 作為總結卡的作答來源——首玩 / 續玩兩者
值相同,重玩時以本輪 `currentChoices` 為準。
原決策(Phase 3 開工時)為固定用 `progress.answers`(「總結卡對應真正被計票的那一票」),
經 code review 對照規格後推翻,理由:(1) 規格書 §6 驗收明寫重玩可看到「**新的**總結卡」;
(2) 規格書 §2.4 驗收「主標區所有比例與答題頁口徑一致」——重玩時答題頁比例是用本輪
`currentChoices` 算的(`selectChoice`),總結卡若用第一輪答案,使用者剛看過的比例與總結卡
對不上,該驗收在重玩情境實質不成立。**防重複計票完全不受影響**:送票與否仍由
`recordAnswer` / `hasAnswered` 依 `progress.answers` 判斷,`progress.answers` 依舊只存
第一次送出統計的選擇、重玩不覆寫(「你上次選的是這個」提示同樣不變)。
勿再改回固定用 `answers`——那會重新違反上述兩條規格驗收。

- `computeQuestionResults(quiz, answers, snapshot)`:逐題套用與 `selectChoice` 相同的
  `computeRatio`(快照缺該題 key 時同樣防呆 `{a:0,b:0}`),回傳
  `{id, index, choice, text, percent, isMinority}`;`percent` 是使用者所選那一邊的百分比,
  `isMinority = percent < 50`(**恰 50% 不算少數派**,規格書 §2.4 定案文字)。
- `deriveIdentityLabel(minorityCount, total)`:身份標籤四檔,用
  `minorityCount / total` 的原始分數比較(不先四捨五入成整數百分比再比對區間),避免
  邊界題數(如 4 題、7 題)在整數化後跟規格的區間文字定義兜不起來:
  - `minorityCount === 0` → 大眾品味代言人
  - `ratio >= 0.5` → 逆流獨行俠
  - `ratio >= 0.25` → 少數派冒險家
  - 其餘(`0 < ratio < 0.25`) → 有主見的主流派
- `pickLoneliestQuestion(results)`:有少數派題時,取其中 `percent` 最低者(用嚴格 `<`
  比較,同率時因迭代順序天然保留題序在前的候選,不需要額外的 tie-break 邏輯);無少數派題
  (含理論上全部恰 50% 的邊界)時,改在全部題目中取 `percent` 最高者(嚴格 `>`
  比較,同率邏輯相同),回傳值多一個 `isFallback: true` 供呼叫端切換文案為「你最大眾的
  一題」,不留空白區塊。
- `computeMatchScore(results)`:所選邊全站比例的算術平均,`Math.round` 取整數。
- `buildShareSummary(minorityCount, total, loneliest)`:組出分享文字,`isFallback` 時用
  「你最大眾的一題有 X% 的人都和你一樣」,否則用「最孤獨的一題只有 X% 的人和你一樣」,
  前綴固定「我在 X/N 題站少數派,…。換你選看看?」——規格書 §2.4 的例句只是示例,兩種
  分支都得包含 X/N 與對應題的百分比(規格書 §2.4 驗收、測試指南 #9)。

### 分享 / 複製連結:`src/lib/share.js`(規格書 §2.4)

**Fallback 矩陣**:

| 情境 | 行為 |
|---|---|
| `navigator.share` 存在 | 點按鈕呼叫 `navigator.share({text, url})`;使用者取消(`AbortError`)或環境限制,呼叫端 `catch` 靜默吞掉,不顯示錯誤(規格未要求) |
| 不存在,`navigator.clipboard.writeText` 可用 | 複製「文字 + 空格 + 連結」,成功回傳 `true` |
| 上者拋錯或不可用(不安全上下文、舊瀏覽器、IG/Threads in-app browser 常見) | fallback 到隱藏 `<textarea>` + `document.execCommand("copy")` |
| 兩層都失敗 | 回傳 `false`,UI 顯示「複製失敗,請手動複製連結」而非「已複製」 |

按鈕文字由 `canUseWebShare()` 決定顯示「分享結果」或「複製連結」(二選一,規格只要求這一顆
按鈕本身要嘛分享要嘛複製,不是三態 UI)。複製成功/失敗皆顯示 2 秒回饋文字
(`.result-share__feedback`),對應測試指南 #10「顯示複製成功回饋」。

**已知限制(本機瀏覽器自動化驗收)**:本機自動化工具(headless Chromium)不支援
`navigator.share`(`typeof navigator.share === "undefined"`),已確認固定落入複製連結分支;
且自動化腳本觸發的 `.click()` 不算瀏覽器認定的「使用者手勢」,`navigator.clipboard.writeText`
在此環境下會被拒絕、`document.execCommand("copy")` fallback 也因此測不出成功結果——這條路徑
本身(呼叫、catch、UI 回饋文字切換)已確認正常運作且不拋錯,但「真的複製成功」與
「`navigator.share` 分支」兩者列為**需人工瀏覽器驗收**(手機瀏覽器/真人點擊)。

### per-quiz OG meta / canonical:`scripts/generate-og-pages.mjs`

**為什麼不能用 JS 動態改 `<head>`**:分享平台的爬蟲(LINE/Threads/FB)與測試指南 #13 指定的
「view-source 檢查」多半不執行 JS,只看伺服器回的原始 HTML;整個網站目前是同一份
`dist/index.html` 的 SPA(`public/_redirects` 的 `/* /index.html 200`),原本每個
`/quiz/:slug` 拿到的都是同一份泛用 head,放不進 per-quiz 的 OG 內容。

**做法**:`npm run build` 新增 `postbuild` hook(`vite build` 之後執行),讀出
`dist/index.html` 的 `<body>`(只有 `<div id="app">`,無 inline 內容)與 build 後帶 hash 的
`<script>`/`<link rel="stylesheet">` 標籤原封複用,對 manifest 內每個題庫產生
`dist/quiz/<id>/index.html`:只有 `<head>` 換成該題庫專屬的
`og:title`/`og:description`/`og:image`(絕對網址)/`og:url`/`canonical`,`<body>` 完全相同,
真人使用者點進去載入的仍是同一份可互動 SPA bundle,答題流程不受影響。**已實測確認**
(`npx wrangler pages dev dist`)Cloudflare Pages 對靜態資產的比對優先於 `_redirects`
規則:請求 `/quiz/demo` 會先 308 轉址到 `/quiz/demo/`(標準靜態網站 pretty-URL 行為),
再拿到 200 與客製 head,不需要改 `public/_redirects`;`/`(入口頁)不受影響,仍走原本的
`dist/index.html`。

### 補充:孤獨稱號選題新規則(2026-07-17 定案,取代單純比 percent 的舊制)

> 疊加在上方 Phase 3.5 之上,不改動 §2.4 擴充 3 條既有驗收記錄(查表邏輯、降級規則、
> 分享文字含稱號的既有格式全部不變),只改「哪一題被選為勳章題」與新增計時句。

**問題**:`pickLoneliestQuestion` 少數派分支永遠取「所選邊比例最低者」,而種子分佈
極端的題(seed 票數兩邊差距懸殊)永遠是 percent 最低者。對 daily-life 16 個稱號做
使用者分佈模擬,結果高度集中:現行規則下前兩個稱號吃掉約 48% 使用者、其餘多個稱號
幾乎無人拿到;改為新規則(選題池內比作答耗時)後模擬最高集中度降到約 12%,分佈明顯
分散。

**新規則**(`src/lib/resultSummary.js` 新增 `pickTitleQuestion(results, times)`,回傳
形狀與 `pickLoneliestQuestion` 相容,含 `isFallback`):

1. 選題池:少數派題(`percent<50`)依 percent 升冪、同 % 依題序排序,取前 3 題
   (不足 3 題取全部)。
2. 池內有計時資料(`times[questionId]` 有值)的題比時間,**最短者得稱號**;
   平手依序比 percent(低者優先)、題序(前者優先)。
3. 池內全部沒有計時資料:fallback 用「現行規則」——pool 已依 percent 升冪排序,
   `pool[0]` 即 percent 最低者。
4. 全站多數派(無少數派題):完全委派 `pickLoneliestQuestion`,mainstream fallback
   行為維持不變,完全不受本次改動影響。

**Fallback 鏈總結**(由上到下,任一層無法決定才落到下一層):
全站多數派 → mainstream fallback(percent 最高者,不變)
有少數派題 → 選題池(前 3 低 percent)內比計時 → 池內無計時 → percent 最低者(池內第一個)。

**計時定義**:「進入題目頁面 → 按下『下一題』」的時間(毫秒,`performance.now()` 差值,
四捨五入到整數 ms)。只有該題「本輪」實際點選過選項才記錄——`quizFlow.js` 的
`goNext(question)` 用 `currentChoices` 是否含該 `question.id` 判斷(目前 UI 下「下一題」
按鈕必須點選才會啟用,此判斷理論上恆為 true,但仍照規格明確判斷,避免未來 UI 改動
後把「已答未點選」的情境誤記成假的短時間)。持久化為 `progress.questionTimes:
{questionId: ms}`,與既有 `answers`/`pendingVotes` 並排存在 localStorage(見
`src/lib/progressStore.js` 的 `createEmptyQuizProgress`/`recordQuestionTime`,
`src/lib/storage.js` 既有的 `{...createEmptyQuizProgress(), ...stored}` merge 邏輯
自動讓舊存檔缺此欄位時降級為空 map,不需額外改動)。無條件覆寫:重玩時若重新點選
則新時間覆蓋舊時間,沒重新點選(不可能發生於目前 UI,見上)則保留舊值。

**「你的孤獨代表作」文案改動**:因新規則選出的題不一定是絕對最低 percent 的那題,
`quizFlow.js` 的 `showComplete()` 把少數派分支(`!loneliest.isFallback`)的
`result-hero__lonely-label` 文字從「最孤獨的一題」改為「**你的孤獨代表作**」;
mainstream fallback 分支的「你最大眾的一題」不變。CSS class 名稱、勳章元件結構
(`result-badge` 系列)完全不動,只改文字內容。

**分享文字加計時句**:`buildShareSummary` 新增第 6 個可選參數 `elapsedMs`(舊呼叫
不帶此參數行為不變)。非 fallback 且 `elapsedMs` 有值且 `<10000`(10 秒)時,在稱號句
的百分比後面加「,這題我 N.N 秒就選了」(毫秒轉秒,`(elapsedMs/1000).toFixed(1)`);
`elapsedMs` 為 `null`/`undefined`、`>=10000`,或 `isFallback` 時維持原句不變。
`quizFlow.js` 傳入 `loneliest.isFallback ? undefined : progress.questionTimes[loneliest.id]`。

**測試**:`tests/resultSummary.test.js` 新增 `pickTitleQuestion` 7 條(池內比時間、
時間平手比 percent、時間與 percent 都平手比題序、池內全無計時 fallback、少數派不足
3 題、選題池只取前 3 低 percent(驗證第 4 題不入池)、全多數派與 `pickLoneliestQuestion`
結果相同)與 `buildShareSummary` 計時句 5 條(<10 秒加句、≥10 秒不加、無計時不加、
fallback 不加、毫秒轉秒邊界值 9999ms→10.0 秒);`tests/progressStore.test.js` 新增
`questionTimes` 預設值與 `recordQuestionTime` 覆寫語意共 3 條。`pickLoneliestQuestion`
本身保留(仍被 `pickTitleQuestion` 內部委派呼叫,獨立測試語意不變,不刪除)。

**裁量決定**:
- 分享文字計時句的錨點文字「只有 X% 和我一樣」只存在於 `buildShareSummary` 的
  「有稱號」分支(`title` truthy);「無稱號」分支(Phase 3 原格式,`只有 X% 的人和你
  一樣`)文字不同,規格範例也只針對有稱號情境,因此本次只在有稱號分支加計時句,
  無稱號分支不動——目前站上唯一上線題庫 daily-life 有 `titles`,demo 已下架,此分支
  差異暫無實際影響。
- `pickTitleQuestion` 的「timed」平手判斷沿用 `pickLoneliestQuestion` 同款「嚴格不等式
  比較」技巧:`timed` 陣列本身沿用 pool 的 percent 升冪/題序排序,只在嚴格更短時間才
  換人,同時間、同 percent 時天然保留先出現(pool 排序在前)者,一次滿足「平手→

**決策**:定稿種子值不能放進 `src/data/quizzes/*.json`——那個目錄會被
`import.meta.glob` 打進前端 bundle(見 `src/lib/quizData.js`),種子票數字屬於「後端
種子資料」不是「題庫內容」,不該讓使用者從瀏覽器 devtools 就看到站方刻意設計的
傾斜分佈。改放 `scripts/seeds/<quizId>.json`(不在任何前端 import 路徑上,只有
Node script 讀取),格式 `{ "<questionId>": { "a": number, "b": number }, ... }`。
`scripts/seeds/daily-life.json` 內容逐題抄自 `subdocs/題庫/daily-life.md > 種子票分佈表`
(單一事實來源在該文件;此檔案只是把表格轉成 script 可讀的 JSON,兩者需保持一致)。

`scripts/seed-votes.mjs` 改法(維持既有 `ON CONFLICT ... DO UPDATE` 可安全重跑語意
不變):對合法清單內每個 `quizId`,先嘗試讀 `scripts/seeds/<quizId>.json`(不存在則
視為 `null`);逐題組值時,該題若在定稿檔中有對應 `questionId` 就用定稿的 `a`/`b`,
否則(定稿檔不存在,或定稿檔存在但缺這一題)fallback 回原本的 20–50 隨機值——
逐題 fallback 而非整庫二選一,對「定稿檔尚未補齊全部題目」的情況更寬容,不會因為
漏了一題就讓整庫種子失效。daily-life 15 題定稿檔涵蓋全部題目,故實際會全數套用
定稿值;尚未建題的美食、宅宅/影視題庫沒有對應定稿檔,繼續吃隨機 fallback。

**測試守護**:`tests/seedVotes.test.js` 斷言 `scripts/seeds/daily-life.json` 的
key 集合與 `src/data/quizzes/daily-life.json` 的 `questionIds` 完全一致(無缺漏、無
多餘)、每題 `a`/`b` 為正整數且 `a + b` 落在合理範圍(daily-life 定稿表逐題皆為
精確 100,故直接斷言 `=== 100`),並逐題核對數值與 `subdocs/題庫/daily-life.md`
定稿表一致(表格內容於測試中原樣抄錄一份作為 oracle,而非重新解析 md 檔——避免
額外引入 markdown parser 這種不成比例的複雜度)。

### 入口頁佔位卡(本子階段,收尾項)

**清單放哪**:`src/lib/placeholderQuizzes.js`(新檔),不是 `home.js` 也不是
`config.js`。理由:清單本身與「產出佔位卡 HTML」都設計成不碰 DOM 的純函式,好處是能
沿用 `src/lib/` 既有慣例(同 `ratio.js`、`recommend.js`)在沒有 jsdom 的測試環境下直接
`import` 測試;不放 `config.js` 是因為這不是 `FEATURE_FLAGS`/門檻那種全站級開關設定,
定位更接近 `home.js` 既有的 `CATEGORY_LABELS`——只是為了讓純函式可測,才多切一個
`lib` 檔而非留在 `home.js` 內。

**資料形狀**:

```js
export const PLACEHOLDER_QUIZZES = [
  { id: "placeholder-food", title: "美食", icon: "🍜" },
  { id: "placeholder-entertainment", title: "娛樂", icon: "🎬" },
];
```

`id` 只供前端渲染(如 `map` 的 key)用,刻意取 `placeholder-` 前綴、不與正式
`manifest.json` 的 `id` 撞名——Phase 6 逐庫上線時新題庫 id 一旦定案,直接從這個常數
陣列刪掉對應項即可,不會有 id 混淆風險。`icon` 為 emoji,搭配 CSS 做封面佔位(見下)。

**渲染**:`renderPlaceholderCard(quiz)` 回傳的 HTML 用 `<div class="quiz-card
quiz-card--placeholder">`,不是 `<a>`,不帶 `href`、不帶 `data-link`——
`src/router.js` 的點擊委派只認 `[data-link]`(見該檔 `startRouter`),沒有這個屬性
天然不會觸發 SPA 導航,不需要額外攔截點擊事件或 `preventDefault`。`home.js` 把這批
卡片渲染在**另一個** `<section class="quiz-list" data-placeholder-list>`,接在
`data-quiz-list` 那個正式題庫卡 section 後面,而不是塞進同一個 section 的陣列裡——
這樣分類 chips(`wireChips`)在 flag on 時重繪 `[data-quiz-list]` 的 `innerHTML`
只會動到正式題庫卡,佔位卡不受影響、永遠留在「正式題庫之後」,不用額外寫「佔位卡
不參與分類篩選」的例外邏輯。CSS 補一條 `.quiz-list + .quiz-list { margin-top: 16px }`
讓兩個相鄰 section 之間維持與同 section 內卡片一致的間距。

**樣式**:封面區沿用 `.quiz-card__cover` 既有的 16:9 比例容器(改
`.quiz-card__cover--placeholder` 疊加 `background: var(--color-muted)` + 置中大字
emoji 取代 `<img>`),版面高度與正式卡一致,不會因為佔位卡穿插其中造成版面跳動。
卡片整體疊加 `opacity: 0.55` + `filter: grayscale(60%)` 做「降飽和 / 半透明」區隔,
另加 `pointer-events: none` 保證滑鼠事件(含 hover)完全不落在佔位卡上,搭配沒有
`href`/`data-link` 形成雙重保險。「N 人玩過」該行不適用,`renderPlaceholderCard`
本來就不產生這行,改用 `.quiz-card__badge` 顯示「建設中」文字標籤。

**測試**:沒有 jsdom,`tests/placeholderQuizzes.test.js` 只測
`src/lib/placeholderQuizzes.js` 匯出的純函式與常數(清單含美食/娛樂、id 不與
manifest 撞名、產出 HTML 不含 `<a>`/`href`/`data-link`/「人玩過」但含「建設中」),
以及讀 `src/data/manifest.json` 斷言佔位卡 id 不在其中(間接涵蓋合法清單 / 種子票 /
OG 產出這些只吃 manifest 的既有管線不受影響,不需要另外跑那幾支 script 才能驗證)。

