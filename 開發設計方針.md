## Phase 1 — 前端答題流程(實作契約)

### 前端選型

**Vanilla JS + Vite**,不上框架(React / Vue 等皆不用)。理由:
- 整個 Phase 1 只是靜態 SPA + mock 資料,頁面數少(入口頁、題庫首頁、答題頁、完成頁佔位),不需要元件框架的複雜度。
- Vite 提供 dev server(HMR)+ build(bundle、hash 檔名、`public/` 靜態複製)剛好打包成 Cloudflare Pages 能直接部署的 `dist/`,設定成本趨近零。
- 頁面渲染採「模板字串 + innerHTML + 事件委派」手刻,不使用 virtual DOM;規模夠小,不會有可維護性問題。

### Feature flag 機制

`src/config.js` 匯出 `FEATURE_FLAGS = { categoryChips: false }`。`home.js` 依此值決定要不要渲染 `.category-chips`(§2.1 驗收第 3 條)。手動驗收時把值暫改 `true`、重整頁面走查,驗完改回 `false`(勿留在 commit 裡)。

### Mock 統計來源

### 瀏覽器返回鍵行為決策

見 `src/router.js` 檔頭註解。摘要:題庫內的「開始作答 → 逐題 → 完成」全程停留在同一個 URL `/quiz/:slug`,題目切換只更新頁面內 JS 狀態,不呼叫 `pushState`/`replaceState`。因此中途按返回鍵會直接離開整個題庫、回到上一個「真正」的 history entry(通常是入口頁 `/`),不會停在題目與題目之間。只有「首頁 → 題庫首頁」這一次導覽會 push 新 entry。已用瀏覽器實測確認(見自我驗收記錄)。

### localStorage 實作決策(§6)

- Key:`wyr:progress`,值為規格書 §6 定義的 `{ [quizId]: { answers, completedAt } }` 結構,一把 key 存全部題庫進度。
- **重玩不覆寫已存答案**:`progressStore.recordAnswer` 對已答過的題目是不動作(no-op),回傳 `isFirstAnswer: false`;`answers` 保存的永遠是「第一次真正送出統計」的選擇,用於顯示「你上次選的是這個」與防止重送 vote。重玩時使用者仍可自由改選,但不影響已存紀錄。
- **`completedAt` 只在第一次完成時寫入**,之後重玩不覆寫(`markCompleted` 對已完成的進度是 no-op),避免重玩洗掉「首次完成時間」這個語意。
- 「你上次選的是這個」提示:單題畫面渲染時,若 `progress.answers[questionId]` 已有值(代表這題在本次以前就答過,只會發生在全部答完後的重玩情境,因為未答完時一律從第一個未答題續玩),在對應選項卡下顯示提示文字。

### 完成頁範圍說明

Phase 1 不含規格書 §2.4 的正式總結卡(比對比例、匹配度、Web Share/複製連結、導流卡屬 Phase 3 範圍)。目前 `quizFlow.js` 的 `showComplete()` 只是一個佔位頁:標題「測驗完成!」、逐題你選了什麼的條列、「再玩一次」/「回首頁」兩個按鈕。用途是讓 §6 的 `completedAt` 寫入邏輯與重玩流程可以被驗收,不是最終 UI。

## Phase 2 — 後端 Workers + D1(實作契約)

### 後端載體:Cloudflare Pages Functions(不是獨立 Worker)

規格書 §1 寫「Cloudflare Workers」,但 Pages Functions 本質上就是跑在同一個 Workers runtime 上的
Worker(Cloudflare 官方文件語:"Functions are just Workers"),差別只在部署與路由慣例。選擇
Pages Functions 而非獨立 Worker 專案,理由:

- **同 repo、同一次部署**:前端(Phase 1 的 Vite 靜態站,產出到 `dist/`)與 API 用同一個
  `wrangler.toml`(`pages_build_output_dir = "dist"`)、同一次 `wrangler pages deploy` 上線,
  Phase 5 不必維護兩個獨立專案與兩份設定。
- **同源、免 CORS**:Pages Functions 與靜態資產同一個 origin 服務,前端 `fetch("/api/...")`
  直接打相對路徑即可,不需處理跨源標頭。
- **本機開發更單純**:`wrangler pages dev` 一個指令就能同時模擬靜態站 + Functions + D1
  綁定,不必另外起一個獨立 Worker dev server 再手動接 D1 binding。
- 檔案結構走 Cloudflare Pages Functions 慣例:`/functions/api/**` 內的檔案依檔名/資料夾自動路由
  (`functions/api/vote.js` → `/api/vote`;`functions/api/stats/[quizId].js` → `/api/stats/:quizId`),
  底線開頭的目錄(`_lib/`)不會被當成路由,用來放共用程式碼。

若日後需要 API 獨立擴充/獨立部署(例如流量大到要跟靜態站分開 scale),可再拆成獨立 Worker,
屆時把 `functions/api/**` 的程式碼原封搬過去即可,邏輯不綁定 Pages 專屬 API。

### wrangler 設定與本機開發方式

`wrangler.toml`(專案根目錄):
```toml
name = "would-you-rather"
compatibility_date = "2026-07-16"
pages_build_output_dir = "dist"

[[d1_databases]]
binding = "DB"
database_name = "would-you-rather-db"
database_id = "<本機佔位值,見檔內註解>"
```
`database_id` 在本機 `--local` 模式下只用來命名 `.wrangler/state` 內的 SQLite 檔,不會連線
Cloudflare 帳號驗證,可以先用隨機 UUID 佔位。**Phase 5 正式部署前**需執行
`wrangler d1 create would-you-rather-db` 取得真正的 database_id 並替換此值,否則
`--remote` / 正式部署會失敗。

本機開發分兩種情境,擇一使用:

1. **只改前端(不需要真 API)**:`npm run dev`(Vite,5173,行為與 Phase 1 相同)。
2. **前端 + API 一起跑(Phase 2 起的預設開發方式)**:兩個終端機視窗:
   - 終端機 A:`npm run dev:worker`(`wrangler pages dev public --port 8788`)——只用
     `public/` 目錄起靜態伺服(內容不影響,實際頁面走終端機 B 的 Vite),重點是掛載
     Functions + 本機 D1 綁定於 8788 埠。
   - 終端機 B:`npm run dev`(Vite,5173,HMR)。`vite.config.js` 設定 `server.proxy`
     把 `/api/*` 轉給 `http://localhost:8788`,瀏覽器只需開 `http://localhost:5173`,
     `fetch("/api/...")` 對前端程式碼而言仍是同源相對路徑,不必處理跨埠 CORS。

   選這個雙終端機方案而非 `wrangler pages dev -- <command>` 的 proxy 子指令:該子指令語法
   在目前 wrangler 版本(4.111)已標示 `[deprecated]`(見 `wrangler pages dev --help`),用
   Vite 官方的 `server.proxy` 是更穩定、不依賴即將移除功能的做法。

生產環境驗證(貼近正式部署行為,單一 origin,無 proxy):
```
npm run build              # 產出 dist/
npx wrangler pages dev dist --port 8788
```
`wrangler.toml` 的 `[[d1_databases]]` 綁定與 `pages_build_output_dir` 皆會自動套用,不需額外
`--d1` 旗標。

### D1 migration

`migrations/0001_init.sql`,schema 照規格書 §5.1 一字不差:
```sql
CREATE TABLE stats (
  quiz_id     TEXT,
  question_id TEXT,
  a_count     INTEGER DEFAULT 0,
  b_count     INTEGER DEFAULT 0,
  PRIMARY KEY (quiz_id, question_id)
);
```
本機套用:`npm run db:migrate`(`wrangler d1 migrations apply would-you-rather-db --local`)。

### 合法清單:build 時從題庫 JSON 產生

`scripts/generate-legal-list.mjs` 讀 `src/data/manifest.json` + `src/data/quizzes/*.json`,
對每個題庫輸出 `{ firstQuestionId, questionIds }`,寫成 `src/data/legalList.generated.json`
(已列入 `.gitignore`,屬 build 產物,不進 repo)。選擇「build 時產生一個靜態 JSON」而非
「Worker 執行期動態讀檔」或「逐檔 dynamic import」,理由:

- Cloudflare Workers runtime 沒有檔案系統,`fs.readdir` 這類在執行期掃描 `quizzes/*.json`
  的做法行不通;而動態組出 per-id 的 `import()` 路徑,esbuild 無法靜態分析、無法可靠 bundle。
- 產生單一靜態 JSON 後,`functions/api/_lib/legalList.js` 只需一行 `import ... from
  "../../../src/data/legalList.generated.json"`,wrangler 在 bundle Functions 時原生支援
  `.json` 的 esbuild loader,一併打包進 Worker,執行期零額外 I/O、零檔案系統依賴。

執行時機:`predev` / `prebuild` npm hook 自動跑(`npm run dev`、`npm run build` 前都會重新產生),
不需手動呼叫;`scripts/seed-votes.mjs` 內部也會先重跑一次確保清單與當下題庫 JSON 同步。

`functions/api/_lib/legalList.js` 匯出 `isLegalQuiz`、`isLegalQuestion`、`getQuestionIds`、
`getFirstQuestionId`、`getAllQuizIds`,`functions/api/vote.js`、`functions/api/stats/[quizId].js`、
`functions/api/played-counts.js` 皆靠這層驗證,拒絕非法 quizId/questionId/choice
(規格書 §5.2)。

### 種子票 script

`scripts/seed-votes.mjs`(`npm run db:seed`):對合法清單內每一題,產生 20–50 的隨機
`a_count`/`b_count`,組成多筆 `INSERT INTO stats (...) VALUES (...) ON CONFLICT(quiz_id,
question_id) DO UPDATE SET a_count = ..., b_count = ...` 寫進暫存 `.sql` 檔,再用
`wrangler d1 execute would-you-rather-db --local --file=<tmp>` 一次性套用,執行後刪除暫存檔。

**可安全重跑**的方式:用 `ON CONFLICT ... DO UPDATE`(不是 `INSERT` 或 `INSERT OR IGNORE`),
重跑不會產生重複 row、不會報 PRIMARY KEY 衝突,只會把該題兩邊計數換成新的一組隨機值——
不是「數值不變」意義上的冪等,而是「結構安全、可重複執行、不累加爆量」。加 `--remote`
旗標可對正式 D1 執行(Phase 5 上線後使用,平時不要對正式庫重跑,會洗掉真實票數)。

### 「N 人玩過」資料來源決策

**決策**:新增 `GET /api/played-counts`,一次回傳所有合法題庫的
`{ quizId: count }`,`count` = 該題庫**第一題**的 `a_count + b_count`。

**理由**:
- 答題流程單向、無回上一題(規格書 §8),第一題是唯一「保證每個開始作答的人都會投的
  一題」,用它的票數總和當「玩過人數」的下限估計,語意上最貼近「N 人玩過」而不需要
  另建一張計數器表或在 `stats` 之外多維護一個 counter row。
- **取捨(已知會灌水)**:種子票（§5.3,每題各自 20–50 隨機)也會讓第一題的種子值計入
  「玩過人數」,新題庫上線初期這個數字不是「真實玩過人次」而是「種子基準 + 真實票數」。
  這與種子票本身「避免 100% vs 0% 尷尬」的取捨精神一致(規格書 §5.3 本來就接受早期
  數字非真實分佈),記錄於此供 Phase 4/5 建題與檢視數字時知悉;若未來要拆分「種子墊底」
  與「真實玩過人次」,需另外加欄位或表,目前判斷不值得為此增加 schema 複雜度。
- **入口頁多題庫 / API 面積最小**:入口頁(`home.js`)對 manifest 內每個題庫各呼叫一次
  `fetchPlayedCount(quizId)`(呼叫端沿用 Phase 1 寫法,不因後端改動而改)。若讓
  `fetchPlayedCount` 直接一對一打 API,題庫一多就會是 N 個請求。改為
  `GET /api/played-counts` 一次回傳全部題庫的值,`src/lib/statsClient.js` 內用一個
  模組層級的快取 `Promise` 把多次呼叫合併成同一個 HTTP 請求(見該檔案 `loadPlayedCounts`),
  呼叫端介面完全不變、API 面積維持在 3 支端點(`GET /api/stats/:quizId`、
  `POST /api/vote`、`GET /api/played-counts`)。
- 題庫首頁(`quizFlow.js`)的 `fetchPlayedCount(slug)` 同樣吃這個快取,不會因為「開始作答」
  前需要顯示「已有 N 人作答」而額外打一次 `GET /api/stats/:quizId`(那支只在按下「開始
  作答」當下呼叫一次,維持 §2.2 驗收)。

### API 契約

**`GET /api/stats/:quizId`**
- 200:`{ [questionId]: { a: number, b: number } }`,涵蓋該題庫合法清單內所有題目;
  尚未有票的題目回傳 `{a:0,b:0}`(不因 D1 缺 row 而漏欄位)。
- 404(非規格書明文要求,實作補充):`quizId` 不在合法清單 → `{ "error": "unknown quizId" }`。

**`GET /api/played-counts`**
- 200:`{ [quizId]: number }`,涵蓋合法清單內所有題庫。

**`POST /api/vote`**
- body:`{ quizId, questionId, choice: "a"|"b" }`。
- 200:`{ "ok": true }`,並以單一 SQL statement
  `INSERT ... ON CONFLICT(quiz_id, question_id) DO UPDATE SET a_count = a_count + 1`
  (或 b_count)原子遞增——SQLite 保證單一 statement 的原子性,並發寫入不掉票。
- 400:body 缺欄位/型別錯誤、`choice` 不是 `"a"`/`"b"`,或 `quizId`/`questionId` 不在
  合法清單 → `{ "error": "..." }`,不寫入 D1。

  不動」的要求,重新命名檔案的效益不足以抵銷改動面。
